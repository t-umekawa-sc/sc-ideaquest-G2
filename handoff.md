## 2. アプリのコンセプト（確定）

- **ideaquest = ゲーム感覚で楽しくアイデアを出し・管理できる WEB アプリ**。テーマはゲーミフィケーション。
- **ターゲット = 企業**（社内でアイデアを集めたい組織）。Docker で動く簡易マルチテナント（会社単位でDB分離）、ロールあり。
- 中心概念:「**クエスト**」= アイデア出しのテーマ単位。カテゴリー・目的/テーマ・期限日・参加メンバー（パーティー）を設定して作成。
- 全体フロー: クエスト作成 → メンバーがアイデア投稿 → 賛成/反対の投票＋アイデア別チャットで議論 → 評価者が5観点で採点・コメント → 集計・選定。
- ゲーミフィケーション: 行動で **XP**（→レベルアップ）と **コイン**（→ショップで装備購入）を獲得し、**3Dアバター**を着せ替え。

## 3. これまでに決定した要件（確定事項）

すべて `doc/要件定義/README.md` に反映済み。

- **ターゲット = 企業**（簡易マルチテナント。ロール: 一般 / クエストグループ管理者 / システム管理者）。
- **クエスト**の入力項目: カテゴリー / 目的・テーマ / 期限日 / 参加メンバー（パーティー）。
- **アイデア**の入力項目: 件名 / タイムリミット / 利害関係者 / アイデア本文 / 価値 / 備考・特記事項 / 関連資料ファイル添付。**必須は 件名・アイデア本文・価値**（他は任意）。
- **アイデアごとにチャットグループ**を自動作成し、メンバーが議論。
- **評価**: 評価者がアイデアを5観点（新規性・影響度・実現度・適合性・コスト）で各5点満点採点＋観点別コメント。
- **評価者の定義**: クエスト作成者＋作成者が評価者権限を付与した参加メンバー（クエスト単位の権限、複数可）。付与対象は参加メンバー限定。
- **クエスト内の権限モデル（6権限）**: 所有者 / クエスト管理 / 評価者 / アイデア投票 / アイデア作成 / コメント作成。
  - 所有者 = 全操作可。**作成者は既定で所有者・剥奪不可**。所有者権限の他者への付与は**作成者のみ**。
  - クエスト管理権限 = クエスト編集（参加メンバー調整・権限付与/剥奪含む）。所有者/管理権限者がその他権限を付与/剥奪できる。
  - 権限を持てるのは参加メンバーのみ。
  - **新規参加メンバーの既定権限 = 投票＋アイデア作成＋コメント**（他は手動付与）。
  - **パーティーから外すと権限は失うが、それまでの入力（アイデア/投票/評価/コメント）は削除せず表示継続**。
- **投票の仕様**: 賛成/反対、1人1票・締切まで変更可。**自分のアイデアにも投票可**。**投票者にXP付与（初期値5・日次上限対象）**。**評価スコアには非影響**（投票と評価は独立）。
- **投票の匿名化（会社ごとON/OFF・既定ON）**: 投票者を公開する記名モードと、集計数のみ表示する匿名モードを**会社（テナント）単位**で切替。**既定=ON（匿名）**。記名モード=賛成/反対のアバターを並べて表示、匿名モード=賛成数/反対数のみ。**投票自体は内部保持**（本人の変更・XP付与に使用、匿名化は表示のみ制御）。
  - **匿名モード時の所有者/管理者への開示**も会社単位で設定（`hide_voters_from_managers`・**既定ON=管理者にも隠す**）。OFFにすると所有者/クエスト管理権限者だけは投票者を確認可（一般には非表示のまま）。
  - 設定は**システム管理者が会社管理の一環で実施**。データ配置: 管理DBの**Companyに `vote_anonymized`（既定ON）／`hide_voters_from_managers`（既定ON）の2フラグ**を持たせ、ログイン時の会社判定と併せて参照。
- **アイデア入力の必須/任意**: **件名・アイデア本文・価値を必須**。タイムリミット・利害関係者・備考・添付は任意。
- **チャット参加範囲**: **パーティーメンバー全員が閲覧・投稿可**（既定でコメント作成権限）。**添付・`@`メンション・通知すべて対応**（通知はMVPアプリ内通知、外部通知は将来）。
- **アイデアの公開範囲**: **そのクエストのパーティー内のみ**（会社全体・グループ全体には非公開。管理者は管理機能の範囲で閲覧）。
- **添付ファイル（アイデア／チャット共通）**: 実体は **S3互換オブジェクトストレージ MinIO（Dockerコンテナ）** に保管し会社ごとにバケット/プレフィックスで論理分離。**物理ファイル名はハッシュ化**（元名で保存しない・パストラバーサル等対策）。**DBはファイルパス＋元アップロードファイル名**（＋サイズ/MIME/アップロード者/日時）を保持。上限サイズ・許可形式の具体値はTBD（初期値設定）。ダウンロードは権限検証＋署名付きURL想定。**compose定義は未作成＝実装フェーズで作成**。
- **ゲーミフィケーション（XP/コイン/レベル）**（数値は初期値・調整可）:
  - **XP付与（成果重視カーブ）**: 自アイデア選定=200 / アイデア投稿=50 / 評価実施=30 / ログイン=10 / 投票=5 / チャット=5。**軽い行動（投票/チャット/ログイン）は1日の付与上限で乱獲対策**。
  - **コイン**: 行動では付与せず、**アイデアの評価点連動でのみ付与（希少）**。全評価者×全5観点の**平均点連動** = `round(平均点×10)`（最大50/アイデア）。投稿者に付与。
  - **レベル**: **総量式の線形上昇**。Lv n→n+1 必要XP = `100+(n-1)×50`（Lv1→2=100、以降+50ずつ）。レベルアップ報酬なし・上限当面なし。
- **5観点の重み付け**: **均等（重みなし）で単純平均**（決定）。コイン付与の平均スコアも同様に均等平均。将来必要なら拡張検討。
- **装備アイテム**（数値は初期値）: **5スロット構成**（頭/顔/体/手持ち/背景・各1点装備）。**入手はコイン購入のみ**（レベル解放・実績報酬なし）。**レアリティ3段階の幅広い価格帯**＝コモン10〜30 / 標準50〜150 / レア300〜800コイン。購入時はサーバー側で残高・価格を検証。
- **3Dアバター**: モデル形式は **VRM**（three-vrm で R3F 連携）。着せ替えは **パーツ差し替え方式**（ベースアバターに5スロットごとの装備メッシュを取り付け・差し替え）。パーツ制作ガイドライン・スキニング詳細はTBD。
- **技術スタック（決定）**: フロント=**Next.js（React）＋Three.js/R3F＋three-vrm** ／ バックエンド=**Python・FastAPI**（ログイン時に管理DBで所属会社判定→会社DBへ動的ルーティング。SQLAlchemy等で接続切替）／ DB=**PostgreSQL**（管理DB1＋会社DB N・全てDockerコンテナ。将来k8s＋CloudNativePGと整合）。
- **提供形態 = Docker コンテナで動く簡易マルチテナント（簡易 SaaS）**。1 デプロイに複数の会社（テナント）が同居。
- **1 デプロイの構成** = Web サーバー(リバプロ)1 ＋ フロント1 ＋ バックエンド1 ＋ 管理DB1 ＋ 会社DB N ＋ **MinIO(S3互換オブジェクトストレージ)1**。バックエンドは1つで、**ログイン時に管理DBで所属会社を判定し、その会社DBへ動的にルーティング**。
- **DB 構成 = 2 層**。①**管理 DB（システム共通・1つ）**＝認証/アカウント/所属会社/クエストグループ所属/会社DB接続情報を一元管理。②**会社 DB（会社ごとに物理分離・別コンテナ/インスタンス）**＝クエスト・アイデア等の機密データを隔離。
- **クエストグループ（会社DB内の中間グルーピング）** = ユーザーは複数所属可。クエストは1つのクエストグループに属す。**表示されるのは「所属グループ内で作られ、かつ自分がパーティー参加中」のクエストのみ**。クエスト作成時に選べるパーティー候補は同一クエストグループ所属者に限定。
- **管理画面 2 階層** = ①システム管理者（運営: 会社作成・DBプロビジョニング・**全アカウント／クエストグループ所属をフル操作**）②クエストグループ管理者（自グループ内アカウントの登録・編集・論理削除のみ）。
- **認証方式 = 独自アカウントで開始**、SSO（SAML/OIDC）連携は将来対応（Could）。**アカウントは管理者発行のみ（自己新規登録なし）／初回ログイン時にパスワード設定**。
- **画面設計の進め方（決定）**: ①`doc/画面設計/画面遷移図.md` で全体フロー合意 → ②画面ごとにワイヤーフレーム → ③**HTML/CSSモック**（`doc/画面設計/mocks/SC-xx_*.html`）→ ④画面設計＝機能詳細（`doc/画面設計/screens/SC-xx_*.md`）。画面IDは SC-00〜SC-91 を採番済み（遷移図参照）。
- **クエストのカテゴリー = 事前定義リストからの選択＋自由入力の併用**。
- **評価「コスト」観点 = 低コストほど高得点**（5点＝非常に低コスト）。5観点の重み付けは**均等（重みなし）で確定**。

## 6. データモデル案（現時点、詳細は README 第9節）

**2層のDBに分かれる**。
- 管理DB（システム共通）: Company（会社/テナント）, Account（ログインID/認証/所属Company/システムロール）, AccountQuestGroup（アカウント×クエストグループ所属）, 運用監査ログ。
- 会社DB（会社ごと物理分離）: User（＋`background_image_path`＝コンテンツ背景画像・任意）/ QuestGroup（クエストグループ）/ QuestGroupMember（User×QuestGroup 多対多）/ Quest（quest_group_id・owner_id・status〔下書き含む〕・`icon_image_path`〔アイコン・任意〕保持）/ QuestMember（パーティー、候補は同一グループ所属者）/ QuestMemberPermission（権限）/ Idea（status〔下書き/公開〕）/ Attachment（パス＋元名・MinIO実体）/ Vote / **Follow（User×Idea アイデアフォロー＝ウォッチ）** / ChatGroup / ChatMessage（添付・メンション）/ Notification（アプリ内通知・フォロー中アイデアの動きも通知）/ Evaluation / EvaluationScore / Item(Equipment) / UserItem / Transaction(Activity)（`quest_id`＝クエスト内ランキング集計用）。
- 添付・各種アップロード画像（背景/クエストアイコン）の実体は MinIO（S3互換オブジェクトストレージ・Dockerコンテナ）に保管。
- **全文検索（FR-31）**: 会社DBの Idea（件名/本文/価値/備考）＋ChatMessage（本文）に日本語全文索引。エンジンは **PGroonga 推奨（要確認）** ／代替 pg_bigm。
- プロビジョニング（確定）: MVPは **docker compose 単一ファイルを手編集**して会社DBを追記（運用コスト最小）。**DB接続情報は .env で管理**。将来は自動化/k8s＋Operatorを検討。
- 未決: テナント数上限・リソース設計、管理DB↔会社DBのUser/クエストグループ所属の同期・配置。
