# コーディング規約

> ideaquest 実装フェーズの共通規約。**実装着手前に骨子を確定**し、必要に応じ追補する（2026-07-21 作成）。
> 仕様本体＝`doc/要件定義/README.md`、データ構造＝`doc/データモデル.md`、API＝`doc/API設計.md`。本ファイルは「どう書くか」の規約。

- 最終更新: 2026-07-21
- 技術スタック（確定）: フロント=**Next.js（React・App Router）＋Three.js/R3F＋three-vrm** ／ バックエンド=**Python・FastAPI** ／ DB=**PostgreSQL（管理DB1＋会社DB N）** ／ オブジェクトストレージ=**MinIO** ／ すべて Docker。

## 1. 最重要原則：レイヤ責務の分離（フロント／バック境界）

**フロントエンドにバックエンド処理（業務ロジック）を実装してはならない。** 業務の判断・確定は必ずバックエンドで行い、フロントは表示・UX・API 呼び出しに徹する。

### 1.1 バックエンド専任（フロント実装禁止）
以下は **FastAPI（サーバー）側でのみ**実装・確定する。フロントに同等ロジックを置かない。

- **認可・権限判定**：システムロール（`system_role`）／クエスト内 6 権限（`permission_type`）の判定、可視範囲（パーティー内のみ）の強制。
- **業務バリデーションの確定**：必須項目・添付 allowlist/サイズ上限(20MB/10件)・会社コード検証・一意制約 等の**権威的検証**。
- **ゲーム計算**：XP 付与と**日次上限**（投票5/チャット10/ログイン1）、レベル算出、**コイン算出 `round(平均点×10)`（最大50）**、SP 付与/消費、実績・ランキング（週起点=月曜0時JST）。
- **状態遷移・冪等**：投票（1人1票・締切判定・初回のみXP）、下書き→公開/投稿、評価の下書き/確定、購入（残高・価格検証）、魔法解放（SP・系統前提）。
- **DB アクセス／MinIO 署名 URL 発行／OTP・セッション／シークレット**：一切をサーバーに閉じる。

### 1.2 フロントエンドの責務
- **表示**と、権限・状態に応じた **UX 上の出し分け**（ボタンの活性/非活性・セクション表示）。
- **API 呼び出し**とレスポンスの描画、フォーム入力補助。
- **クライアント側バリデーションは UX 便宜のみ**であり権威ではない（サーバーが必ず再検証する前提で書く）。エラーは API の `code`／`errors[]`（`doc/API設計.md` §1.7）を正として表示。
- フロントに **DB 接続文字列・シークレット・MinIO 署名鍵・業務しきい値の“正”を持たせない**（しきい値の表示が必要なら API から取得）。

### 1.3 唯一の連携面 = API
- フロント↔バックの連携は **`/api/v1/*`（`doc/API設計.md`）のみ**。フロントは会社DB/管理DBに直接触れない。マルチテナントの会社特定は**セッションの `company_id`**（フロントから `company_id` を送らない＝API設計 §1.5）。

## 2. 共通規約（フロント/バック横断）

- **命名の一貫**：JSON フィールド＝**snake_case**（DB カラムと一致）、URL パス＝**ケバブケース複数形**、Enum 値は**データモデル §3 の値そのまま**（例 `permission_type: "evaluator"`）。API設計 §1.3 と統一。
- **日時**：API 授受は **ISO 8601・UTC**。JST 表示・週起点変換はフロント表示層（集計の週境界判定はサーバー）。
- **ID**：UUID（文字列）。
- **言語/ドキュメント**：仕様・設計ドキュメントは日本語。コード中の識別子は英語、コメントは必要最小限（周辺コードのコメント密度に合わせる）。
- **秘密情報**：接続情報・鍵は `.env`（リポジトリにコミットしない）。`db_identifier`→接続の解決はサーバー（データモデル §4.1）。

## 3. バックエンド（FastAPI）

### 3.1 層構成（Functional Core / Imperative Shell）

**router → application →（domain service / domain model）→ repository** の 4 層。トランザクションと DB アクセスは **application（副作用の殻）に集約**し、**ドメイン層（domain service・domain model）は純粋（I/O・外部参照禁止・引数で全入力を受け取る）** に保つ（2026-07-21 決定）。

| 層 | 責務 | I/O | 備考 |
| --- | --- | --- | --- |
| **router**（presentation） | HTTP 入出力・セッション/認可の依存性解決・リクエスト/レスポンス schema・エラー変換 | HTTP のみ | 業務ロジックを書かない。application を呼ぶだけ |
| **application**（ユースケース） | **エンドポイントと 1:1**。router のデータ⇄ドメインの引数の**変換（DTO マッピング）**、**トランザクション境界（Unit of Work）**、repository で必要データを取得、domain model / domain service（純粋）を呼び出し、結果を永続化し、router 向けに変換して返す | **repository・Tx・MinIO 等の副作用はここに集約** | オーケストレーションの主役（＝副作用の殻） |
| **domain model**（純粋・関数的コア） | 単一集約に属する業務ルール/エンティティ・値オブジェクト。**必要データは全て引数で受け取り**、結果を返す | **禁止（外部参照・DB 一切なし）** | application / domain service から呼ぶ |
| **domain service**（純粋） | 複数モデルにまたがる/どのモデルにも属さない業務ロジック・ポリシー。**同じく引数で受け取り I/O を持たない** | **禁止** | 単一モデルで済むロジックはここに置かず domain model へ |
| **repository**（infrastructure） | DB アクセス（SQLAlchemy）・MinIO 等 | あり | application からのみ呼ぶ |

- **依存方向**：router → application → domain（← repository の結果を application が渡す）。**ドメイン層は repository/HTTP/フレームワークに依存しない**（依存性逆転）。
- **純粋層の型**：domain model / domain service の入出力は dataclass 等の素の値で表し、application が Pydantic/ORM ⇄ ドメイン値の変換を担う。
- **ゲーム計算・ポリシーは domain へ**：XP 付与＋日次上限・コイン算出 `round(平均×10)`・レベル算出・権限判定・評価集計・ランキングスコア・魔法系統前提・投票ルール等は**純粋な domain model / domain service** に置き、DB 非依存でユニットテストする（§3.3）。
- **薄い CRUD の逃げ道**：単純な取得（一覧/単体 GET 等）で業務判断が無い場合、application が repository を直接呼び DTO へ変換してよい（**ドメイン層を無理に通さない**）。domain service を「何でも入れる箱」にしない。
- **application はエンドポイント 1:1**：ユースケースごとに 1 モジュール。複数エンドポイントで再利用する純粋ロジックは domain（model/service）へ切り出す。

### 3.2 スキーマ・その他
- **スキーマ**：リクエスト/レスポンスは **Pydantic モデル**で定義（OpenAPI に反映）。DB モデル（SQLAlchemy）を直接返さない。
- **DB**：SQLAlchemy＋**Alembic マイグレーション**。管理DB／会社DB でエンジンを分け、テナントセッションは依存性 `Depends(get_tenant_session)` で解決（API設計 §1.5）。物理 FK は同一 DB 内のみ（データモデル §2）。
- **監査列**：`created_by_id`/`updated_by_id`/`*_program` 等（データモデル §2.1）は**サーバーがセッションから設定**（クライアント値は無視）。
- **論理削除**：`deleted_at`/`removed_at` 方式。取得系は既定で除外（`deleted_at IS NULL` / `status='active'`）。
- **ユーザ同期**：`accounts` 更新は `account_sync_outbox` INSERT と同一 Tx（API設計 §1.13）。会社DB `users` のミラー列を直接更新しない。
- **エラー**：例外→RFC7807 風レスポンスへ集約変換（API設計 §1.7）。
- **品質ツール**：**ruff（Lint）＋ black（Format）＋ mypy（型）**。型ヒント必須。

### 3.3 テスト（層に対応）
- **domain（model / service）**：純粋ゆえ **DB なしのユニットテスト**（XP日次上限・コイン算出・レベル・権限判定・評価集計・ランキング・魔法系統前提・投票ルール等）。ここを最重点に厚く。
- **application（ユースケース）**：repository をフェイク/インメモリに差し替えたユースケーステスト（オーケストレーション・トランザクション・冪等）。
- **router**：薄いので統合/E2E で最小限。
- ツール：pytest。

## 4. フロント（Next.js）

- **App Router** を採用。**登録・編集・（一覧/ダッシュボードから開く）詳細はモーダル**＝**Parallel Routes（@modal）＋Intercept Routes（(.)）** で URL を持つモーダル（直アクセス/リロードはフルページ）。パネル→モーダルの拡大は **framer-motion `layoutId`**（デザイン標準・SC 各画面）。
- **データ取得**：Server Components / Route Handlers 経由を基本に、業務判断はサーバー(API)に委ねる（フロントで再計算しない）。
- **UI 資産**：デザイン標準（`doc/画面設計/デザイン標準.md`）と共通コンポーネント（`shared.css`／`shared.js` の設計を踏襲）に従う。二層構成（業務層=クリーン／ゲーム層=CRTガラス・1レイヤー1マテリアル）を守る。
- **3D**：three-vrm/R3F。`prefers-reduced-motion` を尊重（エフェクト・演出）。
- **品質ツール**：**ESLint＋Prettier**、TypeScript（型必須）。

### 4.1 フォルダ構成（feature ベース・2026-07-21 決定）

**`app/` はルーティングのみ（薄い）／機能単位は `features/`／機能横断の共有 UI は `components/`**。**依存は一方向**＝ `app → features → components / lib`。**features 同士は原則 import しない**（共有が要れば `components/`（UI）or `lib/`（非UI）へ引き上げる）。`components`・`lib` は下位で features/app に依存しない。

```
frontend/
├── src/
│   ├── app/                      # ルーティングのみ（App Router）。実体は features へ委譲
│   │   ├── layout.tsx  globals.css        # ルートレイアウト＋デザイントークン(現 shared.css 相当)
│   │   ├── (auth)/login/page.tsx          # SC-00 → features/auth
│   │   ├── (app)/                         # 認証後グループ（共通ヘッダーレイアウト）
│   │   │   ├── layout.tsx                 # AppHeader/ベル/ユーザーメニュー/背景
│   │   │   ├── page.tsx                   # SC-01 → features/dashboard
│   │   │   ├── quests/… quests/[questId]/…            # SC-10 / SC-12

│   │   │   ├── shop|avatar|spells|achievements|ranking/…  # SC-30/31/32/40/41

│   │   │   └── (.)quests/create/…  (.)ideas/[id]/edit/…  (.)ideas/[id]/evaluate/…  # Intercept=SC-11/21/25
│   │   └── api/                  # 必要時のみ BFF/Route Handler（薄いプロキシ）
│   │
│   ├── features/                 # ★機能単位（機能スコープの部品・hooks・API呼び出し・型）
│   │   ├── auth/ dashboard/ quests/ ideas/ chat/ evaluations/
│   │   ├── shop/ avatar/ spells/ achievements/ ranking/   # ゲーム層は機能ごとに分割

│   │        ├── components/      # その機能専用UI（例 QuestCard, IdeaVotePanel, ChatComposer）
│   │        ├── hooks/           # その機能専用hooks

│   │        ├── types.ts
│   │        └── index.ts         # 公開API（バレル・外部からはここ経由のみ）
│   │
│   ├── components/               # ★機能横断の共有UI（デザインシステム）
│   │   ├── ui/                   # Button, Card(+accent), Badge, Avatar, Modal, Table,
│   │   │                         #   ListToolbar, Combobox, MultiSelect, Switch, Swatches, Checkbox …

│   │   ├── layout/               # AppHeader, UserMenu, Bell, AppBg, QuestIcon, Poster

│   │   ├── session.ts  date.ts  constants.ts   # セッション取得 / JST・週起点表示変換 / 定数

│   └── types/                    # 横断の型（任意）
├── public/                       # 静的アセット（logo/mascot/login-bg 等・現 mocks/assets）

```

- **`components/` は用途別サブフォルダ**（`ui`/`pixel`/`layout`/`three`）＝二層構成（業務層=クリーン UI／ゲーム層=CRTガラス）を**フォルダでも分離**。現 `shared.css`/`shared.js` の共通部品の移植先。
- **ゲーム層（SC-30/31/32/40/41）は機能ごとに分割**（`shop`/`avatar`/`spells`/`achievements`/`ranking`）。共有する小物（コイン残高表示・レアリティ色）は `components/` へ。
- **`app/api`（BFF）は最小限**。同一オリジンのリバプロ経由で FastAPI `/api/v1` を叩ける前提とし、原則 `lib/api` から呼ぶ（cookie/CSRF はここで一元処理・Server Component からの取得も `lib/api` 経由）。
- **業務ロジック不在の担保（§1）**: features の `api.ts` は「呼ぶだけ」。XP/コイン等の再計算はフロントに書かず、表示に必要な値は API から受け取る。

## 5. Git・進め方

- 要件/設計を 1 つ決めるごとに **README（唯一の要件定義書）＋関連設計ドキュメント＋handoff** を更新し、**ユーザー承認のうえコミット＆プッシュ**。
- **コミット運用（2 段方式）**：変更本体をコミット → handoff の直近コミット一覧に実ハッシュを追記する follow-up コミット（`handoff: 直近コミットのハッシュ参照を修正（<hash>）`）。
- コミットは `main` から作業（必要に応じブランチ）。コミットメッセージ末尾に Co-Authored-By を付す（Claude Code 運用）。
