## 1. 最重要原則：レイヤ責務の分離（フロント／バック境界）

**フロントエンドにバックエンド処理（業務ロジック）を実装してはならない。** 業務の判断・確定は必ずバックエンドで行い、フロントは表示・UX・API 呼び出しに徹する。

### 1.1 バックエンド専任（フロント実装禁止）
以下は **FastAPI（サーバー）側でのみ**実装・確定する。フロントに同等ロジックを置かない。

- **認可・権限判定**：システムロール（`system_role`）／クエスト内 6 権限（`permission_type`）の判定、可視範囲（パーティー内のみ）の強制。
- **業務バリデーションの確定**：必須項目・添付 allowlist/サイズ上限(20MB/10件)・会社コード検証・一意制約 等の**権威的検証**。
- **ゲーム計算**：XP 付与と**日次上限**（投票5/チャット10/ログイン1）、レベル算出、**コイン算出 `round(平均点×10)`（最大50）**、SP 付与/消費、実績・ランキング（週起点=月曜0時JST）。
- **状態遷移・冪等**：投票（1人1票・締切判定・初回のみXP）、下書き→公開/投稿、評価の下書き/確定、購入（残高・価格検証）、魔法解放（SP・系統前提）。
- **DB アクセス／MinIO 署名 URL 発行／OTP・セッション／シークレット**：一切をサーバーに閉じる。

### 1.2 フロントエンドの責務
- **表示**と、権限・状態に応じた **UX 上の出し分け**（ボタンの活性/非活性・セクション表示）。
- **API 呼び出し**とレスポンスの描画、フォーム入力補助。
- **クライアント側バリデーションは UX 便宜のみ**であり権威ではない（サーバーが必ず再検証する前提で書く）。エラーは API の `code`／`errors[]`（`doc/API設計.md` §1.7）を正として表示。
- フロントに **DB 接続文字列・シークレット・MinIO 署名鍵・業務しきい値の“正”を持たせない**（しきい値の表示が必要なら API から取得）。

### 1.3 唯一の連携面 = API
- フロント↔バックの連携は **`/api/v1/*`（`doc/API設計.md`）のみ**。フロントは会社DB/管理DBに直接触れない。マルチテナントの会社特定は**セッションの `company_id`**（フロントから `company_id` を送らない＝API設計 §1.5）。

## 2. 共通規約（フロント/バック横断）

- **命名の一貫**：JSON フィールド＝**snake_case**（DB カラムと一致）、URL パス＝**ケバブケース複数形**、Enum 値は**データモデル §3 の値そのまま**（例 `permission_type: "evaluator"`）。API設計 §1.3 と統一。
- **日時**：API 授受は **ISO 8601・UTC**。JST 表示・週起点変換はフロント表示層（集計の週境界判定はサーバー）。
- **ID**：UUID（文字列）。
- **言語/ドキュメント**：仕様・設計ドキュメントは日本語。コード中の識別子は英語、コメントは必要最小限（周辺コードのコメント密度に合わせる）。
- **秘密情報**：接続情報・鍵は `.env`（リポジトリにコミットしない）。`db_identifier`→接続の解決はサーバー（データモデル §4.1）。

## 3. バックエンド（FastAPI）

- **層構成**：`router`（HTTP 入出力・依存性）→ `service`（業務ロジック・トランザクション）→ `repository`（DB アクセス）。**業務ロジックは service に集約**し、router を薄く保つ。
- **スキーマ**：リクエスト/レスポンスは **Pydantic モデル**で定義（OpenAPI に反映）。DB モデル（SQLAlchemy）を直接返さない。
- **DB**：SQLAlchemy＋**Alembic マイグレーション**。管理DB／会社DB でエンジンを分け、テナントセッションは依存性 `Depends(get_tenant_session)` で解決（API設計 §1.5）。物理 FK は同一 DB 内のみ（データモデル §2）。
- **監査列**：`created_by_id`/`updated_by_id`/`*_program` 等（データモデル §2.1）は**サーバーがセッションから設定**（クライアント値は無視）。
- **論理削除**：`deleted_at`/`removed_at` 方式。取得系は既定で除外（`deleted_at IS NULL` / `status='active'`）。
- **ユーザ同期**：`accounts` 更新は `account_sync_outbox` INSERT と同一 Tx（API設計 §1.13）。会社DB `users` のミラー列を直接更新しない。
- **エラー**：例外→RFC7807 風レスポンスへ集約変換（API設計 §1.7）。
- **品質ツール**：**ruff（Lint）＋ black（Format）＋ mypy（型）**。型ヒント必須。
- **テスト**：pytest。service の業務ルール（XP上限・権限・冪等・コイン算出）を重点的にユニットテスト。

## 4. フロント（Next.js）

- **App Router** を採用。**登録・編集・（一覧/ダッシュボードから開く）詳細はモーダル**＝**Parallel Routes（@modal）＋Intercept Routes（(.)）** で URL を持つモーダル（直アクセス/リロードはフルページ）。パネル→モーダルの拡大は **framer-motion `layoutId`**（デザイン標準・SC 各画面）。
- **データ取得**：Server Components / Route Handlers 経由を基本に、業務判断はサーバー(API)に委ねる（フロントで再計算しない）。
- **UI 資産**：デザイン標準（`doc/画面設計/デザイン標準.md`）と共通コンポーネント（`shared.css`／`shared.js` の設計を踏襲）に従う。二層構成（業務層=クリーン／ゲーム層=CRTガラス・1レイヤー1マテリアル）を守る。
- **3D**：three-vrm/R3F。`prefers-reduced-motion` を尊重（エフェクト・演出）。
- **品質ツール**：**ESLint＋Prettier**、TypeScript（型必須）。

## 5. Git・進め方

- 要件/設計を 1 つ決めるごとに **README（唯一の要件定義書）＋関連設計ドキュメント＋handoff** を更新し、**ユーザー承認のうえコミット＆プッシュ**。
- **コミット運用（2 段方式）**：変更本体をコミット → handoff の直近コミット一覧に実ハッシュを追記する follow-up コミット（`handoff: 直近コミットのハッシュ参照を修正（<hash>）`）。
- コミットは `main` から作業（必要に応じブランチ）。コミットメッセージ末尾に Co-Authored-By を付す（Claude Code 運用）。
