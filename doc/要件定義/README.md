# ideaquest 要件定義

> WEB アプリ「ideaquest」の要件をこのディレクトリで整理していく。
> 本ファイルは全体の目次と概要。詳細は個別ファイルに分割していく。

- 最終更新: 2026-07-13
- ステータス: ドラフト（叩き台）

## 1. プロジェクト概要

| 項目 | 内容 |
| --- | --- |
| プロジェクト名 | ideaquest（アイデアクエスト） |
| 一言でいうと | ゲーム感覚で楽しくアイデアを出し・管理できる WEB アプリ |
| 中心概念 | アイデア出しを「**クエスト**」という単位でまとめる。テーマ・期限・参加メンバー（パーティー）を決めて取り組み、メンバーがアイデアを出し合い、投票・議論・評価する |
| 解決したい課題 | アイデア出し・管理は続けにくく単調になりがち。継続の動機付けが弱い |
| ターゲットユーザー | 企業（社内でアイデアを集めたい組織）。実際に操作するのは企業の従業員と管理者 |
| 提供形態 | **Docker コンテナ**で動作。1 デプロイに複数の会社（テナント）が同居する**簡易マルチテナント（簡易 SaaS）**。**会社ごとに DB を物理分離**（機密保護）し、会社内は**クエストグループ＋パーティー参加**で参照を制限 |
| 提供価値 | ゲーミフィケーション（経験値・レベル・コイン・3Dアバター装備）で、社内のアイデア創出を楽しく活性化・継続できる |

## 2. 背景・目的

企業では社内から広くアイデアを集めたいというニーズがあるが、アイデア出しは地味で継続しづらく、従業員の参加も鈍りがち。
ideaquest は「ゲーム感覚で楽しく操作できる」ことをテーマに、行動（アイデア投稿・ログインなど）に対して
経験値やコインといった報酬を与え、レベルアップやアバターの装飾を通じて、社内のアイデア創出を活性化・習慣化させることを目的とする。
企業単位で導入し、その企業の従業員がユーザーとして参加する形態を想定する。

## 3. スコープ

### やること（In Scope）

- **クエスト**（アイデア出しのテーマ単位）の作成・管理
  - カテゴリー・目的/テーマ・期限日・参加メンバー（パーティー）を指定して作成
- クエストに紐づく **アイデア** の登録・管理（一覧・編集・削除など基本 CRUD）
  - **アイデアの公開範囲 = そのクエストのパーティー内のみ（決定）**。アイデア・投票・評価・チャットの閲覧は当該クエストの参加メンバーに限定（会社全体・グループ全体には公開しない。管理者の閲覧は管理機能の範囲）
- アイデアへの **賛成 / 反対の投票**（参加メンバーによる）
- アイデア単位の **チャットグループ**（メンバー間の意見交換）
  - **リアルタイム配信（決定・2026-07-21）**: チャット（新着/リアクション/魔法/編集/削除）と通知（新着/未読数）を **WebSocket の単一多重接続で即時反映**。**書き込みは REST 維持・WS は配信専用**（受信）、fan-out backbone は Redis Pub/Sub（詳細 `doc/API設計.md` §1.12）。プレゼンス/タイピング表示・外部通知（メール）は将来
- アイデアの **評価**（評価者が 5 観点を 5 点満点で採点＋観点別コメント）
- ゲーミフィケーション要素
  - 経験値（XP）の獲得とレベルアップ
  - コインの獲得
  - ショップでの装備購入
  - 3D アバターへの装備反映
- ユーザー登録・ログイン（ログインでも XP 獲得）
- **多言語対応（i18n・日本語/英語）**：UI 静的文言＋システム生成テキスト（メール/通知/API エラー）＋マスタ表示名（魔法/装備/実績等）を JA/EN 対応。**ユーザー生成コンテンツ（クエスト/アイデア/チャット本文）は翻訳しない**（投稿言語のまま）。**表示言語＝ユーザー設定＋`Accept-Language` フォールバック**（未設定→ブラウザ→既定 `ja`）。実装＝フロント next-intl／バックエンドはロケール対応メッセージ（詳細 `doc/コーディング規約.md` §2.1・`doc/データモデル.md` §8-⑬）
- 会社（テナント）単位の管理（**簡易マルチテナント**）
  - 1 デプロイに複数の会社が同居し、**会社ごとに DB を物理分離**（別 DB コンテナ / インスタンス）してアイデア等の機密情報を隔離
  - **アカウント情報（ログインID・メールアドレス・所属会社）はシステム共通の管理 DB で一元管理**し、ログイン時に所属会社の DB へ接続。**クエストグループ所属は会社 DB（`quest_group_members`）で管理**（機密性重視・二重管理回避。ログイン後の会社 DB 接続時に参照範囲を判定）
- **クエストグループ**（会社 DB 内のグルーピング。ユーザーは複数所属可）による参照制御
  - クエストは 1 つのクエストグループに属し、**表示されるのは「所属クエストグループ内で作られ、かつ自分がパーティーに入っている」クエストのみ**
  - クエスト作成者が選べるパーティーメンバーは**同じクエストグループの所属者に限定**
  - 管理画面 2 階層: **①システム管理（運営: 会社作成・DB プロビジョニング・全アカウント／クエストグループ所属をフル操作）** / **②クエストグループ管理者（自グループ内アカウントの登録・編集・論理削除）**
  - 管理者向けのアイデア閲覧・集計

### やらないこと（Out of Scope）

- 会社をまたいだデータ共有・横断分析（会社間はグループ制限＋DB分離で隔離するため対象外）
- （TBD: 例）多人数リアルタイム共同編集、SNS 連携 等 → 要判断

## 4. ユーザー / ペルソナ

| ペルソナ | 概要 | 主なニーズ |
| --- | --- | --- |
| 一般従業員（パーティーメンバー） | クエストに参加してアイデアを投稿・投票・議論する主役ユーザー | 楽しく手軽にアイデアを出したい・仲間と議論したい・成長やアバターを楽しみたい |
| クエスト作成者（オーナー） | クエストを立ち上げ、テーマ・期限・参加メンバーを設定する | 特定テーマでアイデアを集めたい、進行を管理したい |
| 評価者 | アイデアを 5 観点で採点し、観点別コメントを付ける | 出そろったアイデアを公平に評価・選定したい |
| クエストグループ管理者 | 自分が所属するクエストグループ内のアカウント（従業員）を登録・編集・論理削除する | 自グループのメンバーを管理したい |
| システム管理者（運営） | サービス提供側の全体管理。会社（テナント）の作成・DB プロビジョニング、全アカウントとクエストグループ所属のフル操作 | 会社を安全に追加・分離したい、全体を運用管理したい |

> 補足: 「評価者」はグローバルな固定ロールではなく、**クエスト単位で決まる権限**。
> クエスト作成者は常に評価者となり、加えて作成者が任意のユーザーへ評価者権限を付与できる（第6節フロー詳細参照）。

## 5. 主要ユースケース / ユーザーストーリー

- [ ] クエスト作成者として、カテゴリー・テーマ・期限・参加メンバーを決めてクエストを立ち上げたい。テーマを絞ってアイデアを集めたいから
- [ ] パーティーメンバーとして、参加中のクエストのテーマに沿ったアイデアを登録したい。仲間と成果を出したいから
- [ ] メンバーとして、他メンバーのアイデアに賛成／反対を投票したい。良いアイデアを見極めたいから
- [ ] メンバーとして、アイデアごとのチャットで議論したい。認識をすり合わせてブラッシュアップしたいから
- [ ] 評価者として、各アイデアを 5 観点（新規性・影響度・実現度・適合性・コスト）で採点しコメントしたい。公平に選定したいから
- [ ] ユーザーとして、思いついたアイデアを素早く登録したい。登録すると経験値がもらえて楽しいから
- [ ] ユーザーとして、毎日ログインして経験値を貯めたい。継続の動機になるから
- [ ] ユーザーとして、貯めたコインで装備を買い、3D アバターを着せ替えたい。自分らしくカスタマイズしたいから
- [ ] ユーザーとして、レベルが上がる実感を得たい。成長を感じたいから

## 6. 機能要件

| ID | 機能 | 概要 | 優先度 |
| --- | --- | --- | --- |
| FR-20 | クエスト作成 | 件名・カテゴリー（複数可）・目的/テーマ・期限日・参加メンバー（パーティー）を指定してクエストを作成。**クエストカラー（プリセットパレットから選択・既定色あり）**でカード左帯/アイコンタイル色を設定。**クエストアイコン（画像）を任意設定できる。未設定時は「件名の頭文字＋所有者アバターを重ねた（タイル色＝クエストカラー）」フォールバックを自動表示** | Must |
| FR-21 | クエスト管理 | クエストの一覧・詳細・編集・締切/終了。ステータス管理（募集中/進行中/評価中/完了 等） | Must |
| FR-22 | パーティー編成 | クエストへの参加メンバーの追加・削除 | Must |
| FR-23 | アイデア投票 | 参加メンバーがアイデアに賛成／反対を投票（1人1票・変更可・自分のアイデアも可）。**会社ごとの匿名化設定（既定 ON）**で記名（アバターを並べて表示）／匿名（集計数のみ表示）を切替。匿名時は所有者/管理者への開示可否も会社ごとに設定（既定は非開示）。投票者に XP 付与。**ダッシュボードに未投票アイデアを表示し、賛成/反対の投票ショートカット（カード内クイック投票＋クリックで投票モーダル）を提供** | Must |
| FR-24 | アイデアチャット | アイデアごとにチャットグループを作成し、メンバー間で意見交換（履歴・**ファイル添付・`@`メンション・アプリ内通知**） | Must |
| FR-25 | アイデア評価 | 評価者がアイデアを 5 観点で 5 点満点採点＋観点別コメント（任意）＋**総評（全体コメント・必須）**を登録。**評価結果の公開範囲を指定（既定=パーティー全員/限定）**、**下書き保存可**（本人のみ・確定時に XP/コイン付与）（SC-25） | Must |
| FR-27 | クエスト権限管理 | クエスト単位で参加メンバーに権限（所有者/管理/評価者/投票/アイデア作成/コメント）を付与・剥奪。所有者権限の付与は作成者のみ、作成者の所有者権限は剥奪不可（詳細は第6節「権限モデル」） | Must |
| FR-26 | ファイル添付 | アイデア・チャットに関連資料ファイルを添付・閲覧。実体は MinIO（S3互換）に保管、物理名はハッシュ化、DB はパス＋元ファイル名を保持 | Should |
| FR-01 | アイデア登録 | クエストに紐づくアイデアを新規登録する。登録時に XP・コインを付与 | Must |
| FR-02 | アイデア管理 | 一覧表示・編集・削除 | Must |
| FR-28 | アイデアフォロー | 参加中クエスト内のアイデアをフォロー/解除。**フォロー＝ウォッチ**で、対象アイデアの動き（新規コメント・評価・選定 等）をアプリ内通知。フォロー中アイデアは**ダッシュボードにパネル表示**（クリックで詳細をモーダル表示） | Should |
| FR-29 | 下書き（ドラフト） | クエスト・アイデア・**評価**を**下書き保存**できる。下書きは**本人のみ表示**（公開/投稿/確定するまで非公開）。**XP・コインは公開/投稿/確定時に付与**（下書きでは付与しない）。下書き中のクエスト・アイデア・**評価**は**ダッシュボードの下書きパネルにまとめて表示**し、クリックで該当モーダル（クエスト=SC-11／アイデア=SC-21／評価=SC-25）を開いて続きを編集/採点できる | Should |
| FR-30 | 背景画像のパーソナライズ | 認証済みユーザーが**コンテンツ領域の背景画像を自分でアップロードして設定**できる（**ヘッダーのユーザーメニュー**〔アバターアイコン〕の「背景画像を変更／リセット」）。設定した背景は**その利用者の全認証画面に反映**される（ユーザー個人設定）。可読性のため薄いスクリムを重ねる。画像実体は MinIO に保管（サイズ/形式の上限は初期値設定・TBD） | Could |
| FR-31 | 全文検索（クエスト内） | クエスト詳細に**全文検索タブ**を設け、**アイデアの件名・本文・価値・備考/特記事項・チャット本文・添付ファイル名**を**日本語全文検索**（分かち書き不要）。結果は種別（**アイデア/チャット/添付**）と所属アイデアを示し、**ハイライト付きスニペット**で表示、クリックでアイデア詳細(SC-22)/チャット(SC-24)へ（添付ヒットは紐づく親へ）。検索範囲は当該クエスト（パーティー閲覧範囲）内。エンジンは **PostgreSQL の PGroonga（決定）** | Should |
| FR-32 | チャットリアクション | チャットメッセージに**リアクション**を付けられる。**2層構成**＝①**通常リアクション**（**固定絵文字セット**〔ReactionEmoji マスタ・👍❤️😄🎉🙏👀〕・誰でも無料・同一ユーザー×同一絵文字は不可）②**魔法リアクション**（下記 FR-33 の魔法・特殊エフェクト）。リアクションは集計（種別＋数＋誰が付けたか）で表示。評価スコア・XP には影響しない（社交的演出） | Could |
| FR-33 | スキルポイント & 魔法 | **レベルアップで「スキルポイント（SP）」を獲得**する新通貨。SP を消費して**「魔法/スキル」画面（SC-32）で魔法を解放**（一度きり）。**魔法は系統ごとの段階解放（スキルツリー）**＝上位の解放には**同系統の下位魔法が解放済み**であることが前提。解放した魔法は**チャットのリアクションとして発動**でき、対象メッセージに**永続エフェクト**（例: 炎魔法＝メッセージ枠が燃える演出）を残す。**各魔法は1アイデアチャットにつき1回のみ**（＝「これだ」と思う1メッセージに付ける）。**さらに1メッセージに付けられる魔法は1個まで**（種類を問わず・**他ユーザーが既に魔法を付けているメッセージには付けられない**＝早い者勝ち）。ただし**通常リアクション同様に取消可**で、取り消せば同チャット内の別メッセージに付け替えできる。魔法は**装飾/社交演出のみ**（XP・評価・投票に非影響） | Could |
| FR-34 | アイデア変更履歴・差分 | アイデアの編集を**版（`IdeaRevision`）として記録**し、アイデア詳細（SC-22）の**更新履歴**で日時・編集者・変更フィールド・**差分**（テキストは語句差分、他は旧→新、添付は追加/削除）を表示。**アイデア更新時は投票者＋フォロワーへ通知**。**投票者は自分の投票時点からの差分を確認して賛成/反対を見直せる**（投票は有効のまま・`Vote.voted_revision`/`voted_at` で陳腐化判定・「更新前に投票」表示） | Should |
| FR-35 | 議論アクティビティ可視化 | アイデア詳細（SC-22）に**議論の活発度グラフ**（`ChatMessage` の日次集計・棒グラフ＋参加人数/最終投稿＋活発/落ち着き/停滞ラベル＋アイデア更新マーカー）を表示し、**評価の頃合い**（議論中か収束か）の判断材料にする | Should |
| FR-03 | ユーザー認証 | 独自アカウントでログイン・ログアウト（将来 SSO 連携を見据えた設計）。**全社同一 URL＋ログイン画面で「会社コード」を手入力して会社を特定**（会社選択式にはしない）→ その会社内で**「ログインID」＋パスワード**で認証（ログインID・メールは会社内一意・別カラム）。**会社コード/ID/PW いずれの誤りも一様なエラー**（列挙耐性）。会社コードは端末記憶でプリフィル可。**アカウントは管理者発行のみ・自己新規登録なし**。一般ユーザーは初回ログイン時にパスワード設定 | Must |
| FR-03b | 多要素認証（MFA） | ログインID＋パスワード認証成功後、**登録メールアドレス宛に認証コード（OTP）**を送信し、その入力成功でログインを許可。**「このデバイスを信頼する」ON で次回以降その端末では OTP をスキップ（有効期限30日）／別端末では常に OTP を要求**。**会社ごとに MFA の ON/OFF を設定（既定 ON）**。セッション保持は「デバイス信頼」に一本化（別途の「ログイン状態を保持する」は設けない） | Must |
| FR-04 | 経験値・レベル | 行動に応じて XP を付与し、閾値でレベルアップ | Must |
| FR-05 | ログインボーナス | ログイン（例: 1日1回）で XP を付与 | Should |
| FR-06 | コイン | 行動に応じてコインを付与、残高を管理 | Must |
| FR-07 | ショップ | コインで装備アイテムを購入 | Must |
| FR-08 | 3D アバター | アバター表示、購入した装備の着せ替え・反映 | Must |
| FR-09 | 実績 / バッジ | 達成に応じた実績・バッジ付与と一覧表示（**初期スコープに含む**）。**獲得時にティア（難易度）連動のコインを付与**（ブロンズ◆20/シルバー◆50/ゴールド◆150・初期値）。ティア（銅/銀/金）・数値系は進捗表示・**シークレット実績**あり（達成まで内容を伏せる）（SC-40） | Should |
| FR-10 | ランキング | 社内での XP やレベルのランキング表示（**初期スコープに含む**）。ダッシュボードの**全社 週間ランキング**（週の獲得XP＋コイン）に加え、**クエスト詳細に「クエスト内 週間ランキング」**（そのクエストの活動で得た XP＋コインの合算）を表示 | Should |
| FR-11 | 会社（テナント）管理 | 簡易マルチテナント。会社ごとに DB を物理分離し、ログイン時に所属会社の DB へ接続 | Must |
| FR-12 | 権限・ロール | 一般従業員／クエストグループ管理者／システム管理者などの役割ごとに操作を制御 | Must |

### ゲーミフィケーション設計（決定）

> 方針: **XP は幅広い行動に付与**して参加を促進する（成果行動を厚めに）。**コインは希少通貨**とし、日常の行動では付与せず、**①アイデアの評価点連動（主経路）**と**②実績達成のティア連動ボーナス（一度きり）**でのみ付与する。数値はすべて **初期値（後でバランス調整可）** とする。付与・購入はサーバー側で検証する（第7節）。

#### XP 付与ルール（成果重視カーブ・初期値）

| 行動 | XP | 種別 | 備考 |
| --- | --- | --- | --- |
| 自分のアイデアが最終選定される | 200 | 成果 | クエストで選定されたアイデアの投稿者に付与 |
| アイデア投稿 | 50 | 成果 | 1 アイデアにつき 1 回 |
| 評価実施 | 30 | 成果 | 1 アイデアを評価（採点＋コメント）するごと |
| ログインボーナス | 10 | 日次 | 1 日 1 回まで |
| 投票 | 5 | 参加 | 投票者に付与（評価スコアには非影響）。**各アイデア初回のみ**（賛成/反対の切替・取消・再投票では追加なし） |
| チャット投稿 | 5 | 参加 | 議論への参加を促進 |

- **乱獲対策（決定・初期値）**: 軽い行動は**1 日あたりの付与 XP に上限**を設ける＝**投票 5 回/日・チャット 10 回/日・ログイン 1 回/日**（上限超過分は行動自体は成立するが XP は付与しない）。上限値は初期値・調整可。
- **投票 XP の冪等（決定）**: 投票 XP は**各アイデアにつき初回投票時の 1 回のみ**。賛成→取消→賛成や賛成↔反対の切替では追加付与しない（トグルでの XP 稼ぎ防止）。判定は活動元帳（`reason='vote'` × 対象アイデア）で行い、投票の再作成でも二重付与しない。日次上限は初回投票の回数に対して効く。
- 付与・消費はすべて Transaction/Activity（第9節）に記録し監査可能にする。

#### レベル設計（総量式の線形上昇・初期値）

- **総量式**: レベルは**累積 XP の閾値**で決まる（現在レベルを都度リセットしない）。
- **必要 XP（線形上昇）**: 「レベル n → n+1」に必要な XP = `100 + (n-1) × 50`。
  - 例: Lv1→2 = 100 / Lv2→3 = 150 / Lv3→4 = 200 …（1 レベルごとに +50）。
- レベルアップ時のコイン報酬は設けない（コインは評価連動のみ）。**代わりに、レベルアップごとに「スキルポイント（SP）」を付与**する（下記「スキルポイント & 魔法」）。**レベル上限は設けない（決定・2026-07-20）**＝レベルは累積 XP の進捗表示・過剰 SP は無害に累積（魔法は有限）。

#### スキルポイント & 魔法（決定・数値は初期値）

- **スキルポイント（SP）= レベルアップ報酬の新通貨**。**レベルアップごとに +1 SP**（初期値）。XP/コインとは別残高で保持。
- **魔法の解放**: SP を消費して**「魔法/スキル」画面（SC-32）で魔法を解放**する（**一度解放すれば恒久所有**）。魔法ごとに**SPコスト**を設定（レアリティ的な段階・初期値）。
- **魔法カタログ（初期値・調整可・SC-32）**: レアリティ3段階（コモン/標準/レア＝装備アイテムと共通の色）で 6 種を初期提供。**炎🔥（コモン・1SP・`fire`）／氷❄️（コモン・1SP・`ice`）／雷⚡（標準・2SP・`thunder`）／キラキラ✨（標準・2SP・`sparkle`）／虹🌈（レア・3SP・`rainbow`）／オーラ🌟（レア・3SP・`aura`）**。エフェクトは `spell-fx--*`（shared.css・SC-24 と共通）。SC-32 の各カードで効果プレビューを表示。
- **系統ごとの段階解放（スキルツリー・決定・2026-07-19）**: 魔法は**2系統 × 3段階**（コモン→標準→レア）で構成し、**上位の解放には同系統の下位魔法が解放済みであることを前提**とする。**烈火系**: 炎🔥 → 雷⚡ → 虹🌈／**静輝系**: 氷❄️ → キラキラ✨ → オーラ🌟。起点（コモン）は前提なし。解放は「前提達成 ＋ SP消費」でサーバー検証。
- **魔法の使用＝チャットのリアクション**: 解放済み魔法を、チャットメッセージの魔法リアクションとして発動する。対象メッセージに**永続エフェクト**（例: **炎魔法＝枠が燃える**／氷・雷・キラキラ 等）を付与し、リアクションとして残る（誰が付けたか・数を表示）。
- **使用回数（連打・乱用対策）**: **各魔法は1アイデアチャットにつき1回のみ発動可**（炎・氷・雷…をそれぞれ1回ずつ使える。同じ魔法は2回使えない）。使用自体は無料（解放は一度きりのSP）。**通常リアクション同様にいつでも取消可**で、取り消せば同チャット内の別メッセージへ付け替えできる（＝厳選して付ける体験）。
- **1メッセージ1魔法（決定・2026-07-19）**: **1つのメッセージに付けられる魔法は1個まで**（種類を問わず）。**他ユーザーが既に魔法を付けているメッセージには、他の人は魔法を付けられない**（早い者勝ち・エフェクトの重なり防止）。自分が付けた魔法は取り消して別メッセージへ付け替え可。※**通常リアクション（絵文字）は従来どおり複数・複数人可**（この制限は魔法のみ）。
- **リアクションの2層**: ①**通常リアクション**（絵文字・誰でも無料・エフェクトなし・複数可）②**魔法リアクション**（SP解放・エフェクトあり・各魔法1チャット1回・**1メッセージ1魔法**・取消/付け替え可）。
- **効果範囲**: 魔法・リアクションは**装飾/社交演出のみ**。XP・コイン・評価・投票・ランキングには影響しない。演出は `prefers-reduced-motion` を尊重する。
- 未決（初期値の詳細）: SP付与量/レベルの最終値（**レベル上限は「設けない」で決定**）、**魔法カタログの最終セット**（初期6種を SC-32 で提示済み・スキルツリー化の要否含む）。※**通常リアクションの絵文字セットは ReactionEmoji マスタで確定**（同一ユーザー×同一絵文字は不可＝上限＝セット種類数）。→ 第12節。

#### コイン付与ルール（評価点連動・希少・初期値）

- **コインは行動では付与しない**。**アイデアが評価されたとき、そのアイデアの投稿者に対して、評価点に連動してコインを付与する**（希少通貨）。
- **平均スコア連動**: そのアイデアに付いた**全評価者 × 全 5 観点の平均点（5 点満点）**を基準に付与する。
  - 付与コイン = `round(平均点 × 10)`（平均点は 5 点満点 → 1 アイデアあたり最大 50 コイン）。
  - 例: 平均 4.2 → 42 コイン、平均 2.0 → 20 コイン。
- **確定タイミング＝評価締切で一括確定（決定・2026-07-20）**: 全評価が出そろう締切時に平均を確定して**1 回だけ付与**し、**付与後は取り消さない**（都度再計算は、使用済みコインの残高がマイナスになり得るため不採用）。締切前の下書き/再評価は平均に反映されるが、付与は締切時。
- コインの入手経路は **①評価連動（継続的・主経路）** と **②実績のティア連動ボーナス（一度きり・下記「実績/バッジ」）** の 2 つ。実績報酬は一度きりで乱獲できないため、装備購入の希少性は保つ。

#### 装備 / アイテム（決定・数値は初期値）

- **入手方法（決定）**: **コイン購入のみ**。ショップでコインと引き換えに装備を入手する（装備そのものはレベル解放・実績直接付与などを設けず、**入手経路をコイン購入に一本化**。※実績はコインを付与するが装備を直接付与はしない＝下記「実績/バッジ」）。
- **装備スロット（5 スロット構成・決定）**: アバターは以下 5 スロットを持ち、**各スロット 1 点まで**装備できる。

| スロット | 内容の例 |
| --- | --- |
| 頭 | 髪型・帽子・かぶりもの |
| 顔 | 表情・メガネ・マスク等 |
| 体 | 服・アーマー等 |
| 手持ち | 小道具（武器・ツール等） |
| 背景 | アバター背後の背景 |

- **レアリティと価格帯（3 段階・初期値）**: 各スロットにコモン〜レアの装備を用意する。
  - コモン: 10〜30 コイン（数アイデアの評価で入手できる入門品）
  - 標準: 50〜150 コイン
  - レア: 300〜800 コイン（継続的な貢献で狙う長期目標）
- 未購入の装備もショップで一覧・プレビュー可（TBD: プレビュー方式）。所有済み装備はいつでも付け替え可能。
- **サーバー側検証（第7節）**: 購入時にコイン残高・価格をサーバーで検証し、不正な装備入手を防ぐ。

#### 実績 / バッジ（決定・数値は初期値・FR-09・SC-40）

- **報酬＝ティア連動コイン（決定・2026-07-19）**: 実績を獲得すると**難易度（ティア）に応じてコインを付与**する（**ブロンズ ◆20 / シルバー ◆50 / ゴールド ◆150**・初期値・調整可）。一度きりの恒久ボーナス（グラインド不可）でコインの希少性は保つ。**装備は直接付与しない**（報酬はコインで統一・装備はショップ購入のまま）。※以前の「実績報酬なし」から方針転換。
- **付与**: **サーバーが活動イベント（`Transaction/Activity` 等）から判定**して `UserAchievement` を作成（恒久・冪等）。同時にティア連動コインを付与し `Transaction/Activity` に記録（一度きり検証）。獲得時はアプリ内通知。
- **ティア（難易度の段階）**: ブロンズ / シルバー / ゴールド。**獲得コイン報酬に連動**（◆20/◆50/◆150）。数値系は**進捗（cur/max）**を表示。
- **シークレット実績**: 達成するまで名称・条件・アイコンを伏せる（`？？？`／🔒）。サーバーは未獲得時に内容を返さない。
- **初期カタログ（初期値・調整可・SC-40）**: 投稿（初投稿/10件/50件）・採用（初採用/5件）・評価（10件/50件）・投票（初投票/50回）・議論（コメント10/100）・継続（連続ログイン7日/30日）・成長（Lv5/Lv10）・魔法（初解放/全解放）・装備（初購入/全購入）・シークレット（深夜投稿 ほか）。詳細は `doc/画面設計/screens/SC-40_実績バッジ.md`。

### クエスト → アイデア → 評価 のフロー詳細

全体の流れ:

```
クエスト作成（テーマ・期限・パーティー）
   → メンバーがアイデア投稿
   → メンバーが賛成/反対を投票 ＋ アイデア別チャットで議論
   → 評価者が各アイデアを 5 観点で採点・コメント
   → 結果を集計・選定
```

#### クエスト内の権限モデル

権限は**クエスト単位**で管理し、対象は**そのクエストの参加メンバー（パーティー）に限定**する（＝権限を持つには参加メンバーであることが前提）。1 メンバーは複数の権限を併せ持てる。

| 権限 | できること | 付与できる人 | 備考 |
| --- | --- | --- | --- |
| 所有者権限 | クエストの**全操作**（下記すべてを含む） | **クエスト作成者のみ**が他ユーザーへ付与できる | クエスト作成者は既定で所有者。**作成者の所有者権限は解除不可**。他ユーザーへの付与により所有者は複数存在しうる |
| クエスト管理権限 | クエストの編集（参加メンバーの追加/削除、各権限の付与/剥奪 を含む） | 所有者 / クエスト管理権限者 | 所有者権限の付与だけは作成者のみ（上記） |
| 評価者権限 | アイデアに対する評価（採点・コメント）の入力 | 所有者 / クエスト管理権限者 | — |
| アイデア投票権限 | アイデアへの投票（賛成/反対） | 所有者 / クエスト管理権限者 | — |
| アイデア作成権限 | アイデアの作成 | 所有者 / クエスト管理権限者 | — |
| コメント作成権限 | アイデアに紐づくグループチャットでのコメント | 所有者 / クエスト管理権限者 | — |

補足・ルール:
- 所有者権限は他のすべての権限を包含する（所有者は評価・投票・作成・コメント・管理すべて可能）。
- **所有者権限の他ユーザーへの付与はクエスト作成者のみ**が行える。その他の権限（管理/評価/投票/作成/コメント）は、所有者またはクエスト管理権限者が付与・剥奪できる。
- クエスト作成者の所有者権限は剥奪・委譲による喪失ができない（常に所有者）。
- 権限付与の対象は参加メンバーのみ。メンバーをパーティーから外すと、そのクエストで保持していた権限も失う（＝以降は投票/評価/作成/コメント/管理ができない）。ただし**そのメンバーがそれまでに入力した内容（アイデア・投票・評価・コメント）は削除せず、表示されたまま残す**。
- **新規参加メンバーの既定権限**: パーティー参加時に **アイデア投票権限・アイデア作成権限・コメント作成権限** を自動付与する。所有者/クエスト管理/評価者の各権限は既定では付与せず、必要に応じて所有者・管理権限者が付与する。

#### 添付ファイルの扱い（決定・アイデア添付／チャット添付 共通）

- **保管先**: **S3 互換オブジェクトストレージ（MinIO）を Docker コンテナで構成**し、1 デプロイ内に含める。会社（テナント）ごとにバケット/プレフィックスで論理分離する。会社 DB には実体ファイルを保存しない。
- **物理ファイル名のハッシュ化（決定）**: ストレージ上の物理オブジェクト名は**ハッシュ等で生成した非可読名**とし、**アップロードされた元のファイル名のままでは保存しない**（名前衝突・情報漏えい・パストラバーサル対策）。
- **DB が保持する情報（決定）**: 会社 DB の Attachment レコードには **①ストレージ上のファイルパス（物理オブジェクトキー）** と **②アップロード時の元ファイル名（表示・ダウンロード時に使用）** の両方を保持する。加えてサイズ・MIME タイプ・アップロード者・アップロード日時等を保持。
- **上限サイズ / 形式**: 1 ファイルの上限サイズと許可形式（画像・PDF・Office 文書 等）を設ける（**具体値は TBD・初期値として設定**）。アップロードはサーバー側でサイズ・MIME を検証する。
- **アクセス制御**: ダウンロードは所属会社・参照権限（当該クエストのパーティー参加）を検証したうえで、署名付き URL 等の一時アクセスで提供する（実装方式の詳細は TBD）。

#### 投票

- 対象: アイデア投票権限を持つ参加メンバー（パーティーメンバー）
- 選択肢: **賛成 / 反対**（1人1票、締切まで変更可能を想定）
- **自分のアイデアへの投票も許可**する
- **投票の匿名化（会社ごとに ON/OFF・決定）**: 投票者を公開するか匿名にするかを、**会社（テナント）単位の設定**で切り替える。設定は会社内の全クエスト・全投票に一律適用する。**既定値は ON（匿名モード）**。
  - **記名モード（匿名化 OFF）**: 誰が賛成/反対したかを公開する。アイデアごとに **賛成したユーザーのアバター** と **反対したユーザーのアバター** を並べて表示（賛成数・反対数もあわせて表示）。
  - **匿名モード（匿名化 ON・既定）**: 投票者の identity を他ユーザーに表示せず、**賛成数・反対数（集計）のみ**を表示する。個々の投票者のアバター・氏名は出さない。ただし**投票者本人には自分の投票内容を表示**し、締切まで変更できる。
  - **匿名モード時の所有者/管理者の扱い（会社ごとに ON/OFF・決定）**: 匿名モードでも**所有者権限・クエスト管理権限を持つユーザーには投票者を開示するか**を会社単位で設定する。
    - **ON（既定）**: 所有者/管理者にも投票者を開示せず、集計数のみ表示（全員に対して匿名を徹底）。
    - **OFF**: 匿名モードでも所有者/管理者だけは投票者（賛成/反対したユーザー）を確認できる（一般メンバーには非表示のまま）。
    - ※この設定は匿名モード（匿名化 ON）のときのみ意味を持つ。記名モードでは全員が投票者を確認できる。
  - どちらのモードでも投票は内部的に記録し（1人1票の制約・XP 付与・変更履歴に使用）、匿名化は**表示上の扱い**のみを制御する。
  - 設定は **システム管理者（運営）が会社の作成・管理の一環として設定**する。
- **XP**: 投票したユーザーに XP（初期値 5）を付与する。軽い行動のため**日次上限の対象**。ただし **評価者による評価には影響しない**（投票結果と評価スコアは独立。第6節ゲーミフィケーション設計参照）。匿名モードでも XP 付与は同じ（付与は本人にのみ紐づく内部処理）

#### チャット

- アイデアごとに 1 つのチャットグループを自動作成
- **参加範囲（決定）**: そのクエストの**パーティーメンバー全員**が閲覧・投稿できる（コメント作成権限を持つメンバー。既定で全参加メンバーに付与）
- メッセージ投稿・履歴表示
- **ファイル添付（決定）**: チャットメッセージにファイルを添付できる。保管・命名方式はアイデア添付と共通（後述「添付ファイルの扱い」参照）
- **メンション（決定）**: `@ユーザー` でパーティーメンバーをメンションできる
- **通知（決定）**: メンション・自分のアイデアへの新規コメント等を通知する。**MVP はアプリ内通知**（ベル/未読）から開始。メール等の外部通知は将来対応（TBD）
- チャット投稿には XP を付与する（軽い行動として日次上限の対象。第6節ゲーミフィケーション設計参照）

#### 評価（評価者による採点）

1 つのアイデアに対し、評価者が以下 5 観点を **各 5 点満点** で採点し、**観点ごとにコメント** を記載する。

| 観点 | 説明 | 配点 |
| --- | --- | --- |
| 新規性 | アイデアの目新しさ・独自性 | 1〜5 |
| 影響度 | 実現した場合のインパクトの大きさ | 1〜5 |
| 実現度 | 実現可能性の高さ | 1〜5 |
| 適合性 | クエストのテーマ・目的への合致度 | 1〜5 |
| コスト | コスト面（**低コストほど高得点**。5点＝非常に低コスト） | 1〜5 |

- 各観点に採点値 + コメント（テキスト）を保持
- 合計/平均点でアイデアを比較・ランキング。**5 観点は均等（重み付けなし・決定）** で単純平均する（コイン付与の平均スコアも同様に均等平均。第6節）。将来重み付けが必要になれば拡張を検討
- **評価者（決定）**: クエストごとに以下のユーザーが評価できる。
  - **クエスト作成者**（デフォルトで評価者）
  - **クエスト作成者が評価者権限を付与したユーザー**（クエスト単位で付与・剥奪）
  - → 評価者は 1 クエストに複数存在しうる（＝1アイデアに複数評価者の評価が付きうる）
- 評価者権限を付与できる対象は **そのクエストの参加メンバーに限定（決定）**。詳細は上記「クエスト内の権限モデル」を参照
- **評価結果の公開範囲（決定・2026-07-18）**: 評価者は評価の入力（SC-25）時に、その評価結果の**公開範囲を指定できる**。
  - **既定＝パーティー全員に公開**（当該クエストの参加メンバーが結果〔観点別スコア・平均・観点別コメント・**総評**〕を閲覧可）。フィードバックを全員で共有し学びにする狙い。
  - **限定に切替可**: 対象は**投稿者＋評価者本人＋所有者/クエスト管理権限者**。範囲外メンバーへの見せ方（集計〔平均〕のみ表示 or 完全非表示）は TBD（第12節）。
  - データ配置: `Evaluation.visibility`（`party`〔既定〕/`limited`）。閲覧可否はサーバー側で最終制御。
- **評価の下書き（決定・2026-07-18）**: 評価も**下書き保存**できる（`Evaluation.status`＝`draft`/`submitted`）。
  - **下書き＝評価者本人のみ表示**（他の評価者・パーティーには非公開）。全観点そろっていなくても保存でき、後で再開できる。
  - **XP（評価実施）・投稿者コインは「確定（submitted）」時のみ付与**（下書きでは付与しない）。確定は全 5 観点の採点＋総評を必須とする。
  - **評価の下書きはダッシュボードの下書きパネルに表示**（クエスト/アイデアの下書きと同じ最上段パネル）。クリックで SC-25 を開いて採点を再開（FR-29・SC-01）。

## 7. 非機能要件

- **性能**: 3D 表示を含むため、描画パフォーマンスに配慮（TBD: 目標 FPS / 初回ロード時間）
- **可用性**: （TBD）
- **セキュリティ**: 認証情報の保護、コイン・装備の不正取得防止 → 付与・購入はサーバー側で検証
- **認証・MFA**: パスワードはハッシュ保存・HTTPS 前提。**メールOTP**（6桁・有効期限10分・単回使用・試行/再送制限・レート制限）。認証失敗理由は詳細に返さない（アカウント存在推測の防止）。**信頼デバイストークンは暗号乱数・ハッシュ保存の HttpOnly/Secure Cookie**（有効期限30日、パスワード変更/リセット等で失効）。**ログインIDとメールアドレスは別項目**（ログインIDは非メールの識別子も可）。残存リスク: OTP 配送がメールのため、登録メール侵害はログイン突破の一要因になり得る（将来 TOTP 追加を検討）。**メール送信基盤（SMTP/送信サービス）は TBD**
- **テナント分離 / 機密保護**: アイデアは会社の機密情報。**会社ごとに DB を物理分離**し、ログインユーザーは所属会社のクエストのみ参照可（クロステナントアクセス禁止）。会社 DB への接続はログイン時に確定した所属会社に限定
- **スケーラビリティ（決定・2026-07-20・初期目安）**: 会社追加は DB コンテナのプロビジョニングで対応。**MVP は単一ホスト Docker Compose で〜10 社をソフト上限**（各社 1 PostgreSQL コンテナ・0.5vCPU/512MB〜1GB 目安）。**会社 DB ごとに小さなコネクションプール（or PgBouncer 等）**で接続爆発を抑制。会社数/DBサイズ/接続数/CPU を監視し、閾値超過で**水平分割（ホスト追加）or Kubernetes＋Postgres Operator（CloudNativePG）へ移行**（自動プロビジョニングも同段階）。数値は初期目安・調整可
- **アクセシビリティ / 対応ブラウザ**: 3D（WebGL）対応ブラウザ前提。非対応環境の扱いは TBD

## 9. データモデル

（主要エンティティの案。**ER 図・テーブル定義・型・制約・インデックスの詳細は `doc/データモデル.md` を参照**。本節は正典として値集合・要件を保持し、詳細実装は同ファイルに展開する）

> **DB 構成（2 層）**: エンティティは 2 種類の DB に分かれて格納される。
> - **管理 DB（コントロールプレーン / システム共通・1 つ）**: 全社横断で必要な認証・振り分け情報を持つ。
> - **会社 DB（データプレーン / 会社ごとに物理分離・別コンテナ）**: 各社のクエスト・アイデア等の機密データを隔離して持つ。
> User の識別情報はログイン振り分けのため管理 DB に置き、会社 DB 側にはプロフィール/XP 等の業務データを持つ（同期方法の詳細は TBD）。

### 管理 DB（システム共通）

- **Company（会社 / テナント）**: 名称、**会社コード（`company_code`＝業務上の対人向け会社識別コード・全社で一意。`db_identifier` とは別項目。半角英大文字/数字/ハイフン・4〜20文字・大小文字を区別せず大文字正規化・原則作成時に確定し以後不変）**、接続先会社 DB の識別子（接続情報の実体は `.env` で管理し、キー/参照のみ保持）、状態、**会社カラー（`color`＝プリセットパレットの hex・クエストカラーと同方式）**、**会社アバター/アイコン（`icon_image_path`＝MinIO のオブジェクトキー・任意。未設定時は「会社名の頭文字＋会社カラー」タイルで表示。クエストアイコンと同方式）**、**投票匿名化フラグ（vote_anonymized: ON/OFF・会社単位・既定 ON）**、**匿名モード時に所有者/管理者へ投票者を隠すフラグ（hide_voters_from_managers: ON/OFF・会社単位・既定 ON）**、**MFA有効フラグ（mfa_required: ON/OFF・会社単位・既定 ON）**、作成日時。※各フラグはログイン時に会社判定と併せて参照
- **Account**: ログインアカウント。**ログインID（`login_id`＝ログイン用の識別子・**会社内で一意**。メールアドレスとは別項目）**、**メールアドレス（`email`＝OTP 送信・初回パスワード設定リンク・将来の外部通知の宛先。**会社内で一意**）**、認証情報、所属 Company、システムロール（システム管理者／クエストグループ管理者／一般）、**表示言語（`locale`＝`ja`/`en`・未設定は Accept-Language→既定でフォールバック・プロフィール編集で変更可・会社DBの User へミラー・OTP/通知メールの言語にも使用）**。※**ログインIDとメールは明確に別カラムで保持**（ログインIDは英数字等の任意識別子でも可、メール変更でログインIDは不変）。**`login_id`・`email` はユーザー本人のプロフィール画面から変更可**（会社内一意を検証。認証に使うため源泉は管理DBの Account）
  - **会社DBへのミラー（性能対策）**: ユーザ一覧を会社DB単独で高速描画するため、`login_id`/`email`/`status`（有効/無効）/`password_set`（PW設定済み）/`system_role`（システムロール）/`last_login_at`（最終ログイン）/`locale`（表示言語）を会社DBの User へ**一方向ミラー複製**（源泉＝管理DBの Account）。反映はアウトボックス方式（アカウント発行/編集/無効化/PWリセット・プロフィール編集を契機に冪等・結果整合で同期）。一覧は「有効なユーザのみ」を会社DBで絞り込み、API 往復を行わない
- **TrustedDevice**: MFAの信頼済み端末（Account × 端末）。`account_id / token_hash（Cookie値のハッシュ） / label（UA由来など） / created_at / last_used_at / expires_at（発行から30日） / revoked`。次回ログイン時にパスワード成功後へ照合し、有効なら OTP をスキップ。パスワード変更/リセット・ユーザーの全端末サインアウト・管理者操作で失効
- **OTPチャレンジ**: メール認証コード。`account_id / code_hash / purpose / expires_at（10分） / attempts / consumed`。Redis 等の一時ストア推奨（DB化も可）
- **SystemAdmin 操作ログ**: 会社作成・DB プロビジョニング等の運用監査（案）
- **AccountSyncOutbox**: `login_id`/`email`/`status`/`password_set`/`system_role`/`last_login_at`/`locale` を会社 DB の User へ一方向ミラーするためのアウトボックス（管理DB。§8-① 決定）
- ※**クエストグループ所属（QuestGroupMember）は会社 DB 側で管理**（管理 DB には持たない＝§8-① 決定・機密性重視）。割当はシステム管理者がフル操作、クエストグループ管理者は自グループ範囲のみ。**グループ内ロール（`quest_group_role`＝`admin`/`member`）の `admin` 付与/剥奪はシステム管理者のみ**（QG 管理者はメンバー発行/編集は可だが `admin` 昇格は不可・2026-07-20 決定）。**所属解除は論理削除（`removed_at` トゥームストーン）で監査保持**し、アカウント無効化（`status`）とは独立に扱う

> クエストグループの所属（QuestGroupMember）は**会社 DB 側で管理**する（§8-① 決定）。参照範囲判定はログイン後の会社 DB 接続時に会社 DB を引いて行うため、管理 DB に所属情報を持たない（二重管理回避・機密性重視。全社横断のQG参照は当面対象外）。

### 会社 DB（会社ごとに分離）

- **User**: ユーザー（会社 DB 内。Account と対応）。ロール（一般／評価者／クエストグループ管理者）、XP、レベル、コイン残高、**スキルポイント残高（`skill_point_balance`＝レベルアップで付与・魔法解放に消費）**、**コンテンツ背景画像（`background_image_path`＝MinIO のオブジェクトキー・任意。個人設定・全認証画面に適用）**、**表示言語（`locale`＝Account からのミラー・`ja`/`en`）**を保持
- **QuestGroup（クエストグループ）**: 会社 DB 内のグルーピング。名称、作成日時。ユーザーは複数所属可
- **QuestGroupMember**: User × QuestGroup の所属（多対多）。パーティー候補・クエスト参照範囲の基礎
- **Quest**: クエスト。**所属クエストグループ（`quest_group_id`）**、作成者（`owner_id`＝既定で所有者・剥奪不可）、件名（名前）、**カラー（`color`＝クエストカラー・プリセットパレットのhex/enum）**、**カテゴリー（複数可＝別テーブル/配列で保持。事前定義＋自由入力）**、目的/テーマ、期限日、**ステータス（下書き / 募集中 / 進行中 / 評価中 / 完了。`下書き`＝作成者本人のみ表示・公開前）**、**アイコン画像（`icon_image_path`＝MinIO のオブジェクトキー・任意。未設定時は「件名の頭文字＋所有者アバター」で描画）**、作成日時、更新日時。**削除は論理削除（`deleted_at`＋`deleted_by_id` トゥームストーン・物理削除しない／所有者・管理権限者）**。※公開（下書き→募集中）時に XP を付与
- **QuestMember（パーティー）**: クエスト参加メンバー（User × Quest の多対多、参加日時を保持）。権限を持つ前提となる所属。**候補は当該クエストの所属クエストグループのメンバーに限定**
- **QuestMemberPermission**: クエスト参加メンバーが持つ権限（QuestMember に対し多）。権限種別（所有者/クエスト管理/評価者/投票/アイデア作成/コメント）、付与者、付与日時。※作成者の所有者権限は剥奪不可
- **Idea**: アイデア。所属クエスト、投稿ユーザー、件名、本文（アイデア）、タイムリミット、**利害関係者（複数可＝別テーブル/配列。事前定義＋自由入力）**、価値、備考、**ステータス（下書き / 公開。`下書き`＝作成者本人のみ表示・投稿前）**、作成日時、更新日時（※会社 DB 内のため会社は暗黙）。**削除は論理削除（`deleted_at`＋`deleted_by_id` トゥームストーン・物理削除しない／投稿者本人・クエスト管理権限者。FR-02 の CRUD「削除」）＝従属〔チャット/投票/評価/版/添付〕は監査保持**。※投稿（下書き→公開）時に XP・コインを付与
- **Attachment**: 添付ファイル（アイデアまたはチャットメッセージに紐づく）。**ストレージ上のファイルパス（物理オブジェクトキー＝ハッシュ名）** と **元のアップロードファイル名**、サイズ、MIME タイプ、アップロード者、アップロード日時を保持。実体は MinIO（S3 互換）に保管し会社ごとに論理分離
- **Vote**: アイデアへの投票（User × Idea）。種別（賛成／反対）、投票日時、**投票時点の版（`voted_revision`）・`voted_at`**（最新版と比較して「更新前に投票」＝陳腐化を判定し、SC-22 で差分見直し導線を出す）。1人1票（ユニーク制約）。投票者は常に内部保持（本人の変更・XP 付与に使用）。自分のアイデアへの投票も可。**表示上の記名/匿名は Company の匿名化フラグに従う**（匿名化 ON では他ユーザーに投票者を表示せず集計数のみ）
- **IdeaRevision**: アイデアの変更履歴（版）。所属 Idea、編集者、作成日時、**変更内容（フィールド単位の差分 or スナップショット）**、変更メモ（任意）。SC-22 の更新履歴/差分に使用（FR-34）。編集（保存）ごとに 1 版を追加し、投票者＋フォロワーへ通知
- **ChatGroup**: アイデアごとのチャットグループ（Idea と 1:1）
- **ChatMessage**: チャット投稿。所属 ChatGroup、投稿者、本文、投稿日時、**添付（Attachment への参照・複数可）**、**メンション対象ユーザー**。**編集は本人のみ・履歴は残さない**（「編集済み」表示）。**削除は論理削除（トゥームストーン）**で**本人＋クエスト管理権限者/QG管理者/システム管理者**が可（削除者・日時を記録）。**クエスト完了後は投稿/編集/削除/リアクションを凍結**（読み取り専用）
- **Reaction**: チャットメッセージへのリアクション（ChatMessage × User）。種別（`normal`＝絵文字／`magic`＝魔法）、絵文字コード or **Spell 参照**（魔法時）、作成日時。集計表示（種別＋数＋付与者）に使用。**装飾/社交演出のみ**（XP/評価/投票に非影響）。**魔法リアクションの制約**＝①1 アイデアチャット（ChatGroup）につき「当該ユーザー × 同一 Spell」1 件まで（ユニーク制約 `chat_group_id × user_id × spell_id`）②**1 メッセージ（ChatMessage）につき魔法は 1 件まで**（`type=magic` は `message_id` ユニーク＝**先に付いていれば他ユーザーは付けられない**）。取消/付け替え可。通常リアクション（`normal`）はこの制限対象外（複数・複数人可）だが**同一ユーザー×同一絵文字は不可**。通常リアクションの絵文字は**固定セット**（下記 **ReactionEmoji** マスタ・任意ピッカーにしない）
- **ReactionEmoji**: 通常リアクションの絵文字マスタ（会社DB・シード配布）。`code`（安定キー）/絵文字/ラベル/並び順/有効フラグ。**追加はシード更新のみ**（全社同一・`is_active` で有効化/無効化）。初期6種＝👍いいね/❤️ありがとう/😄わらい/🎉おめでとう/🙏感謝/👀確認
- **Spell（魔法マスタ）**: 解放可能な魔法。名称、アイコン、**エフェクト種別（例: `fire`／`ice`／`thunder`／`sparkle`／`rainbow`／`aura`）**、**SPコスト**、レアリティ段階、**系統（`line`＝烈火系/静輝系 等）**、**前提魔法（`requires`＝同系統の1つ下位の Spell・起点は null）**、説明（数値は初期値）。**上位の解放は前提魔法の解放済みが必要（スキルツリー）**
- **UserSpell**: ユーザーが解放済みの魔法（User × Spell・ユニーク）。解放日時。入手＝SP消費のみ（恒久所有）
- **Notification**: ユーザーへの通知（アプリ内・**一覧は SC-02**）。宛先 User、種別（メンション／自アイデアへの新規コメント／**フォロー中アイデアの新規コメント・評価・選定**／**投票済み・フォロー中アイデアの更新（版追加・FR-34）**／**実績獲得（コイン付与）**／**魔法リアクション受領** 等）、参照先（Idea/ChatMessage/IdeaRevision/Achievement 等）、既読フラグ、作成日時。**MVP はアプリ内のみ**（外部通知は将来）
- **Follow（アイデアフォロー）**: User × Idea のフォロー関係（多対多・ユニーク制約）。作成日時。**フォロー＝ウォッチ**（対象アイデアの動きを通知）。フォロー可能な対象は「自分がパーティー参加中のクエストのアイデア」に限定（アイデアの公開範囲に従う）。ダッシュボードのフォローパネルの表示源
- **Evaluation**: アイデアへの評価（Idea × 評価者）。評価者、評価日時、**総評（`overall_comment`＝アイデア全体への総括コメント・確定時必須／下書きは空可）**、**状態（`status`＝`draft`〔本人のみ・付与なし〕/`submitted`〔確定・XP/コイン付与〕）**、**公開範囲（`visibility`＝`party`〔既定・パーティー全員に公開〕/`limited`〔投稿者＋評価者＋所有者/管理のみ〕）**。閲覧可否はサーバー側で最終制御
- **EvaluationScore**: 評価の観点別スコア（Evaluation に対し多）。観点（新規性/影響度/実現度/適合性/コスト）、点数（1〜5）、コメント
- **Item / Equipment**: 装備アイテム。名称、**スロット（頭/顔/体/手持ち/背景）**、**レアリティ（コモン/標準/レア）**、価格（コイン）、**装備パーツ（VRM 用メッシュ）の参照**（パーツ差し替え方式）
- **UserItem**: ユーザーが所有・装備中のアイテム（多対多）。所有フラグ＋スロットごとの装備中フラグ（各スロット 1 点）。入手経路はコイン購入のみ
- **Transaction / Activity**: XP・コイン・**スキルポイント（SP）**の付与・消費履歴（不正防止・監査用、案）。**関連クエスト（`quest_id`・任意）を保持**し、週次のクエスト内ランキング集計（そのクエストの活動で得た XP＋コイン）に利用（SP はランキング対象外）
- **Achievement（実績マスタ）**: 実績。カテゴリ、ティア（銅/銀/金）、アイコン、名称、説明、達成条件、数値目標（任意）、**シークレットフラグ**（達成まで内容を伏せる）、**報酬コイン（`coin_reward`＝ティア連動 ◆20/◆50/◆150・初期値）**
- **UserAchievement**: ユーザーが獲得した実績（User × Achievement）。獲得日時、数値系の進捗（cur/max）。付与はサーバーが活動から判定・冪等
- ※**マスタ表示名の多言語化（i18n・§8-⑬）**: 全社同一シードのマスタ（**Spell / Item / Achievement / ReactionEmoji**）の**表示名・説明・ラベルは JA/EN の対で保持**（`name_ja`/`name_en` 等）。API が解決済みロケールに応じて返す。Enum 値・`code`・`condition`（判定ロジック）はコードなので不変

## 10. 技術スタック（決定）

- **フロントエンド（決定）**: **Next.js（React）**。3D 表示に **Three.js / React Three Fiber（R3F）**、VRM は **three-vrm** で連携。React エコシステムで R3F/VRM との相性を優先。
  - **モーダルUI（決定）**: 登録・編集・（一覧/ダッシュボードから開く）詳細表示は **App Router の Parallel Routes（`@modal` スロット）＋ Intercept Routes（`(.)` 等）** でモーダル差し込みを実装。URL を持つモーダル（共有・戻る/進むで開閉・リロードでフルページ）を標準とする。パネル→モーダルの**拡大アニメーションは共有要素アニメーション（framer-motion の `layoutId`）**で実装（デザイン標準のマスコット同様、画面作成フェーズで作り込み）。
- **3D アバター（決定）**: モデル形式は **VRM** を採用（人型アバター特化・ボーン/表情が標準化され着せ替えに有利。**three-vrm** で R3F と連携）。
  - **着せ替え方式（決定）**: **パーツ差し替え方式**。ベースアバターに対し、5 スロット（頭/顔/体/手持ち/背景）ごとの装備パーツ（メッシュ）を取り付け・差し替えする。パーツ追加で装備を拡張しやすい構成とする。
  - 装備パーツは各スロット共通のリグ/取り付け基準に合わせて用意する（TBD: パーツの制作ガイドライン・スキニング方式の詳細）。
- **バックエンド（決定）**: **Python / FastAPI**。軽量・高速で、将来の AI/分析連携にも拡張しやすい。**ログイン時に管理 DB で所属会社を判定し、当該会社 DB へ動的にルーティング**（SQLAlchemy 等でリクエストごとに接続先エンジン/セッションを切り替える構成）。
- **セッション/リアルタイム基盤（決定）**: **Redis** を 1 デプロイに含める＝①セッション/OTP の TTL ストア（§8・API設計 §1.4）②**リアルタイム配信の fan-out backbone（Pub/Sub）**。**リアルタイムは WebSocket の単一多重接続**（チャット/通知の即時反映・**書き込みは REST 維持で WS は配信専用**・API設計 §1.12）。
- **データベース（決定）**: **PostgreSQL**。管理 DB 1 ＋ 会社 DB N を、いずれも Docker コンテナで構成。将来の Kubernetes ＋ Postgres Operator（例: CloudNativePG）移行方針とも整合。
  - **日本語全文検索（決定）**: **PGroonga**（Groonga ベースの PostgreSQL 拡張）。**日本語の分かち書き不要**・ランキング（`pgroonga_score`）・スニペット/ハイライト（`pgroonga_snippet_html`）を **DB 内で完結**でき、追加コンテナが不要。**拡張は会社DBのみに導入**（検索は会社スコープ・管理DBは不要）。索引対象＝会社DBの **Idea（件名/本文/価値/備考）＋ ChatMessage（本文）＋ Attachment（添付ファイル名）**＝結果種別「アイデア/チャット/添付」。代替＝**pg_bigm**（bigram・軽量・不採用）。将来の高精度化（あいまい検索・シノニム）が必要なら **Meilisearch / OpenSearch＋kuromoji**（別コンテナ・データ同期が必要）を検討。※Postgres イメージに PGroonga を同梱する（カスタムイメージ）。
- **インフラ / ホスティング**: **Docker コンテナ**前提。1 デプロイ内に「Web サーバー（リバプロ・**WebSocket Upgrade 許可**）＋フロント＋バックエンド（API＋outbox ワーカ）＋管理 DB（システム共通）＋会社ごとの DB コンテナ（物理分離）＋**Redis（セッション/OTP/Pub/Sub）**＋**S3 互換オブジェクトストレージ（MinIO）**」を構成。
  - **オブジェクトストレージ（決定）**: 添付ファイルの実体を保管する **MinIO コンテナ**を 1 デプロイに含める。会社（テナント）ごとにバケット/プレフィックスで論理分離し、物理オブジェクト名はハッシュ化する（第6節「添付ファイルの扱い」参照）。※compose 定義は実装フェーズで作成する（現時点で compose ファイルは未作成）。
  - **プロビジョニング方式（MVP・確定）**: **docker compose の単一ファイルを手編集**して会社ごとの DB サービスを追記する（運用コスト最小を優先）。会社追加は「compose に DB サービス追記 → `up` → 管理 DB に接続先を登録」の手動オペレーションとなる。
  - **DB 接続情報（確定）**: `.env` ファイルで管理し、compose から参照（接続情報はリポジトリにコミットしない）。
  - **将来**: テナント数増加・可用性が必要になった段階で、Docker Engine API による自動プロビジョニングや **Kubernetes ＋ Postgres Operator（例: CloudNativePG）** への移行を検討。
- **API 設計（決定・詳細は `doc/API設計.md`）**: **REST / JSON ＋ OpenAPI**（FastAPI 自動生成の OpenAPI をソースオブトゥルース）。ベース `/api/v1`、JSON=snake_case、認証＝会社コード→login_id＋PW→（信頼端末なら OTP スキップ）→メール OTP→httpOnly Cookie セッション、会社DB は**セッションの `company_id` から動的ルーティング**（クロステナント禁止）。エラーは RFC7807 風。
- **レイヤ責務の分離（決定・詳細は `doc/コーディング規約.md`）**: **フロントエンドにバックエンド処理（業務ロジック）を実装しない**。認可・業務バリデーションの確定・ゲーム計算（XP/コイン/SP・日次上限）・状態遷移/冪等・DB/MinIO/認証は**バックエンド専任**、フロントは表示・UX 出し分け・API 呼び出しのみ（クライアント側検証は権威にしない）。連携面は `/api/v1/*` のみ。

## 13. 用語集

| 用語 | 定義 |
| --- | --- |
| ideaquest | 本プロジェクトの名称 |
| 会社（テナント） | データ分離の単位。1 デプロイに複数同居し、会社ごとに DB を物理分離 |
| クエストグループ | 会社 DB 内のグルーピング。ユーザーは複数所属可。クエストはいずれかのグループに属し、パーティー候補・参照範囲の基礎になる |
| 管理 DB（コントロールプレーン） | システム共通の DB。認証・アカウント・所属会社・クエストグループ所属・会社 DB 接続情報を一元管理 |
| 会社 DB（データプレーン） | 会社ごとに物理分離した DB（別コンテナ/インスタンス）。クエスト・アイデア等の機密データを隔離 |
| オブジェクトストレージ（MinIO） | 添付ファイルの実体を保管する S3 互換ストレージ（Docker コンテナ）。会社ごとにバケット/プレフィックスで論理分離し、物理ファイル名はハッシュ化する |
| システム管理者 | サービス運営側。会社の作成・DB プロビジョニング、全アカウントとクエストグループ所属をフル操作 |
| クエストグループ管理者 | 自分が所属するクエストグループ内のアカウントを登録・編集・論理削除できる管理者 |
| クエスト | アイデア出しのテーマ単位。カテゴリー・目的/テーマ・期限・参加メンバーを持つ |
| パーティー / パーティーメンバー | クエストに参加するメンバーの集まり・各メンバー |
| 投票 | アイデアに対する賛成／反対の意思表示 |
| 投票匿名化 | 投票者を表示するか（記名モード）匿名にするか（匿名モード＝集計数のみ表示）を切り替える設定。**会社（テナント）単位**で ON/OFF する（既定 ON）。匿名モード時に所有者/管理者へ投票者を開示するかも会社単位で別途設定できる（既定は非開示） |
| チャットグループ | アイデアごとに作られる、メンバー間の意見交換の場 |
| 評価者 | アイデアを 5 観点で採点・コメントする役割のユーザー（評価者権限を持つ参加メンバー） |
| 評価観点 | 新規性・影響度・実現度・適合性・コストの 5 項目（各 5 点満点） |
| 所有者権限 | クエストの全操作が可能な権限。作成者は既定で所有者（剥奪不可）。他者への付与は作成者のみ |
| クエスト管理権限 | クエストの編集（参加メンバー調整・権限付与含む）が可能な権限 |
| XP（経験値） | 行動に応じて付与されるポイント。累積でレベルが上がる |
| レベル | XP 累積に応じたユーザーの成長度合い |
| コイン | ショップで装備を購入するための通貨 |
| 装備 / アイテム | 3D アバターに着せ替えできる要素 |
| アバター | ユーザーを表す 3D キャラクター |
