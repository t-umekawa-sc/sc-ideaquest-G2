## 1. プロジェクト概要

| 項目 | 内容 |
| --- | --- |
| プロジェクト名 | ideaquest（アイデアクエスト） |
| 一言でいうと | ゲーム感覚で楽しくアイデアを出し・管理できる WEB アプリ |
| 中心概念 | アイデア出しを「**クエスト**」という単位でまとめる。テーマ・期限・参加メンバー（パーティー）を決めて取り組み、メンバーがアイデアを出し合い、投票・議論・評価する |
| 解決したい課題 | アイデア出し・管理は続けにくく単調になりがち。継続の動機付けが弱い |
| ターゲットユーザー | 企業（社内でアイデアを集めたい組織）。実際に操作するのは企業の従業員と管理者 |
| 提供形態 | **Docker コンテナ**で動作。1 デプロイに複数の会社（テナント）が同居する**簡易マルチテナント（簡易 SaaS）**。**会社ごとに DB を物理分離**（機密保護）し、会社内は**クエストグループ＋パーティー参加**で参照を制限 |
| 提供価値 | ゲーミフィケーション（経験値・レベル・コイン・3Dアバター装備）で、社内のアイデア創出を楽しく活性化・継続できる |

## 2. 背景・目的

企業では社内から広くアイデアを集めたいというニーズがあるが、アイデア出しは地味で継続しづらく、従業員の参加も鈍りがち。
ideaquest は「ゲーム感覚で楽しく操作できる」ことをテーマに、行動（アイデア投稿・ログインなど）に対して
経験値やコインといった報酬を与え、レベルアップやアバターの装飾を通じて、社内のアイデア創出を活性化・習慣化させることを目的とする。
企業単位で導入し、その企業の従業員がユーザーとして参加する形態を想定する。

## 3. スコープ

### やること（In Scope）

- **クエスト**（アイデア出しのテーマ単位）の作成・管理
  - カテゴリー・目的/テーマ・期限日・参加メンバー（パーティー）を指定して作成
- クエストに紐づく **アイデア** の登録・管理（一覧・編集・削除など基本 CRUD）
  - **アイデアの公開範囲 = そのクエストのパーティー内のみ（決定）**。アイデア・投票・評価・チャットの閲覧は当該クエストの参加メンバーに限定（会社全体・グループ全体には公開しない。管理者の閲覧は管理機能の範囲）
- アイデアへの **賛成 / 反対の投票**（参加メンバーによる）
- アイデア単位の **チャットグループ**（メンバー間の意見交換）
- アイデアの **評価**（評価者が 5 観点を 5 点満点で採点＋観点別コメント）
- ゲーミフィケーション要素
  - 経験値（XP）の獲得とレベルアップ
  - コインの獲得
  - ショップでの装備購入
  - 3D アバターへの装備反映
- ユーザー登録・ログイン（ログインでも XP 獲得）
- 会社（テナント）単位の管理（**簡易マルチテナント**）
  - 1 デプロイに複数の会社が同居し、**会社ごとに DB を物理分離**（別 DB コンテナ / インスタンス）してアイデア等の機密情報を隔離
  - **アカウント情報（ログインID・所属会社・所属クエストグループ）はシステム共通の管理 DB で一元管理**し、ログイン時に所属会社の DB へ接続
- **クエストグループ**（会社 DB 内のグルーピング。ユーザーは複数所属可）による参照制御
  - クエストは 1 つのクエストグループに属し、**表示されるのは「所属クエストグループ内で作られ、かつ自分がパーティーに入っている」クエストのみ**
  - クエスト作成者が選べるパーティーメンバーは**同じクエストグループの所属者に限定**
  - 管理画面 2 階層: **①システム管理（運営: 会社作成・DB プロビジョニング・全アカウント／クエストグループ所属をフル操作）** / **②クエストグループ管理者（自グループ内アカウントの登録・編集・論理削除）**
  - 管理者向けのアイデア閲覧・集計

### やらないこと（Out of Scope）

- 会社をまたいだデータ共有・横断分析（会社間はグループ制限＋DB分離で隔離するため対象外）
- （TBD: 例）多人数リアルタイム共同編集、SNS 連携 等 → 要判断

## 4. ユーザー / ペルソナ

| ペルソナ | 概要 | 主なニーズ |
| --- | --- | --- |
| 一般従業員（パーティーメンバー） | クエストに参加してアイデアを投稿・投票・議論する主役ユーザー | 楽しく手軽にアイデアを出したい・仲間と議論したい・成長やアバターを楽しみたい |
| クエスト作成者（オーナー） | クエストを立ち上げ、テーマ・期限・参加メンバーを設定する | 特定テーマでアイデアを集めたい、進行を管理したい |
| 評価者 | アイデアを 5 観点で採点し、観点別コメントを付ける | 出そろったアイデアを公平に評価・選定したい |
| クエストグループ管理者 | 自分が所属するクエストグループ内のアカウント（従業員）を登録・編集・論理削除する | 自グループのメンバーを管理したい |
| システム管理者（運営） | サービス提供側の全体管理。会社（テナント）の作成・DB プロビジョニング、全アカウントとクエストグループ所属のフル操作 | 会社を安全に追加・分離したい、全体を運用管理したい |

> 補足: 「評価者」はグローバルな固定ロールではなく、**クエスト単位で決まる権限**。
> クエスト作成者は常に評価者となり、加えて作成者が任意のユーザーへ評価者権限を付与できる（第6節フロー詳細参照）。

## 5. 主要ユースケース / ユーザーストーリー

- [ ] クエスト作成者として、カテゴリー・テーマ・期限・参加メンバーを決めてクエストを立ち上げたい。テーマを絞ってアイデアを集めたいから
- [ ] パーティーメンバーとして、参加中のクエストのテーマに沿ったアイデアを登録したい。仲間と成果を出したいから
- [ ] メンバーとして、他メンバーのアイデアに賛成／反対を投票したい。良いアイデアを見極めたいから
- [ ] メンバーとして、アイデアごとのチャットで議論したい。認識をすり合わせてブラッシュアップしたいから
- [ ] 評価者として、各アイデアを 5 観点（新規性・影響度・実現度・適合性・コスト）で採点しコメントしたい。公平に選定したいから
- [ ] ユーザーとして、思いついたアイデアを素早く登録したい。登録すると経験値がもらえて楽しいから
- [ ] ユーザーとして、毎日ログインして経験値を貯めたい。継続の動機になるから
- [ ] ユーザーとして、貯めたコインで装備を買い、3D アバターを着せ替えたい。自分らしくカスタマイズしたいから
- [ ] ユーザーとして、レベルが上がる実感を得たい。成長を感じたいから

### ゲーミフィケーション設計（決定）

> 方針: **XP は幅広い行動に付与**して参加を促進する（成果行動を厚めに）。**コインは希少通貨**とし、行動では付与せず、**アイデアの評価点に連動**してのみ付与する。数値はすべて **初期値（後でバランス調整可）** とする。付与・購入はサーバー側で検証する（第7節）。

#### XP 付与ルール（成果重視カーブ・初期値）

| 行動 | XP | 種別 | 備考 |
| --- | --- | --- | --- |
| 自分のアイデアが最終選定される | 200 | 成果 | クエストで選定されたアイデアの投稿者に付与 |
| アイデア投稿 | 50 | 成果 | 1 アイデアにつき 1 回 |
| 評価実施 | 30 | 成果 | 1 アイデアを評価（採点＋コメント）するごと |
| ログインボーナス | 10 | 日次 | 1 日 1 回まで |
| 投票 | 5 | 参加 | 投票者に付与（評価スコアには非影響） |
| チャット投稿 | 5 | 参加 | 議論への参加を促進 |

- **乱獲対策（決定）**: チャット投稿・投票・ログインなど軽い行動は、**1 日あたりの付与 XP に上限**を設ける（連投による XP 稼ぎを抑制）。上限値は初期値として設定し調整可。上限を超えた分の行動自体は成立するが XP は付与しない。
- 付与・消費はすべて Transaction/Activity（第9節）に記録し監査可能にする。

#### レベル設計（総量式の線形上昇・初期値）

- **総量式**: レベルは**累積 XP の閾値**で決まる（現在レベルを都度リセットしない）。
- **必要 XP（線形上昇）**: 「レベル n → n+1」に必要な XP = `100 + (n-1) × 50`。
  - 例: Lv1→2 = 100 / Lv2→3 = 150 / Lv3→4 = 200 …（1 レベルごとに +50）。
- レベルアップ時の追加報酬（コイン等）は設けない（コインは評価連動のみ）。レベル上限は当面設けない（TBD: 必要なら上限設定）。

#### コイン付与ルール（評価点連動・希少・初期値）

- **コインは行動では付与しない**。**アイデアが評価されたとき、そのアイデアの投稿者に対して、評価点に連動してコインを付与する**（希少通貨）。
- **平均スコア連動**: そのアイデアに付いた**全評価者 × 全 5 観点の平均点（5 点満点）**を基準に付与する。
  - 付与コイン = `round(平均点 × 10)`（平均点は 5 点満点 → 1 アイデアあたり最大 50 コイン）。
  - 例: 平均 4.2 → 42 コイン、平均 2.0 → 20 コイン。
- 複数評価者・追加評価で平均点が変動した場合は**最新の平均点に基づく付与額との差分**を調整する（TBD: 確定タイミングを「評価締切時に一括確定」とするか「都度再計算」とするかは実装時に整理）。
- コインの主な入手経路はこの評価連動のみとし、装備購入の希少性を担保する。

#### 装備 / アイテム（決定・数値は初期値）

- **入手方法（決定）**: **コイン購入のみ**。ショップでコインと引き換えに装備を入手する（レベル解放・実績報酬などは設けず、入手経路をコインに一本化してコインの価値を明確化）。
- **装備スロット（5 スロット構成・決定）**: アバターは以下 5 スロットを持ち、**各スロット 1 点まで**装備できる。

| スロット | 内容の例 |
| --- | --- |
| 頭 | 髪型・帽子・かぶりもの |
| 顔 | 表情・メガネ・マスク等 |
| 体 | 服・アーマー等 |
| 手持ち | 小道具（武器・ツール等） |
| 背景 | アバター背後の背景 |

- **レアリティと価格帯（3 段階・初期値）**: 各スロットにコモン〜レアの装備を用意する。
  - コモン: 10〜30 コイン（数アイデアの評価で入手できる入門品）
  - 標準: 50〜150 コイン
  - レア: 300〜800 コイン（継続的な貢献で狙う長期目標）
- 未購入の装備もショップで一覧・プレビュー可（TBD: プレビュー方式）。所有済み装備はいつでも付け替え可能。
- **サーバー側検証（第7節）**: 購入時にコイン残高・価格をサーバーで検証し、不正な装備入手を防ぐ。

### クエスト → アイデア → 評価 のフロー詳細

全体の流れ:

```
クエスト作成（テーマ・期限・パーティー）
   → メンバーがアイデア投稿
   → メンバーが賛成/反対を投票 ＋ アイデア別チャットで議論
   → 評価者が各アイデアを 5 観点で採点・コメント
   → 結果を集計・選定
```

#### クエスト内の権限モデル

権限は**クエスト単位**で管理し、対象は**そのクエストの参加メンバー（パーティー）に限定**する（＝権限を持つには参加メンバーであることが前提）。1 メンバーは複数の権限を併せ持てる。

| 権限 | できること | 付与できる人 | 備考 |
| --- | --- | --- | --- |
| 所有者権限 | クエストの**全操作**（下記すべてを含む） | **クエスト作成者のみ**が他ユーザーへ付与できる | クエスト作成者は既定で所有者。**作成者の所有者権限は解除不可**。他ユーザーへの付与により所有者は複数存在しうる |
| クエスト管理権限 | クエストの編集（参加メンバーの追加/削除、各権限の付与/剥奪 を含む） | 所有者 / クエスト管理権限者 | 所有者権限の付与だけは作成者のみ（上記） |
| 評価者権限 | アイデアに対する評価（採点・コメント）の入力 | 所有者 / クエスト管理権限者 | — |
| アイデア投票権限 | アイデアへの投票（賛成/反対） | 所有者 / クエスト管理権限者 | — |
| アイデア作成権限 | アイデアの作成 | 所有者 / クエスト管理権限者 | — |
| コメント作成権限 | アイデアに紐づくグループチャットでのコメント | 所有者 / クエスト管理権限者 | — |

補足・ルール:
- 所有者権限は他のすべての権限を包含する（所有者は評価・投票・作成・コメント・管理すべて可能）。
- **所有者権限の他ユーザーへの付与はクエスト作成者のみ**が行える。その他の権限（管理/評価/投票/作成/コメント）は、所有者またはクエスト管理権限者が付与・剥奪できる。
- クエスト作成者の所有者権限は剥奪・委譲による喪失ができない（常に所有者）。
- 権限付与の対象は参加メンバーのみ。メンバーをパーティーから外すと、そのクエストで保持していた権限も失う（＝以降は投票/評価/作成/コメント/管理ができない）。ただし**そのメンバーがそれまでに入力した内容（アイデア・投票・評価・コメント）は削除せず、表示されたまま残す**。
- **新規参加メンバーの既定権限**: パーティー参加時に **アイデア投票権限・アイデア作成権限・コメント作成権限** を自動付与する。所有者/クエスト管理/評価者の各権限は既定では付与せず、必要に応じて所有者・管理権限者が付与する。

#### クエストの入力項目

| 項目 | 説明 | 必須 |
| --- | --- | --- |
| カテゴリー | クエストの分類。**事前定義リストからの選択＋自由入力の併用**（定義済みカテゴリを基本とし、なければ自由入力で追加できる） | ○ |
| 目的・テーマ設定 | クエストで何を達成したいか、どんなアイデアを募るか | ○ |
| 期限日 | アイデア募集/クエストの締切日 | ○ |
| 参加メンバー（パーティー） | クエストに参加するメンバー（複数） | ○ |

#### アイデアの入力項目

| 項目 | 説明 | 必須 |
| --- | --- | --- |
| 件名 | アイデアのタイトル | ○ |
| タイムリミット | アイデアの想定期限（実施/検討の締切） | 任意 |
| 利害関係者 | このアイデアに関係する人・部署 | 任意 |
| アイデア | アイデア本文（内容の説明） | ○ |
| 価値 | このアイデアがもたらす価値・メリット | ○ |
| 備考 / 特記事項 | 補足事項 | 任意 |
| 関連資料ファイル添付 | 参考資料の添付（複数可） | 任意 |

> 必須は **件名・アイデア本文・価値** の 3 項目（決定）。投稿ハードルを下げつつ、評価に必要な「価値」は必須とする。

#### 添付ファイルの扱い（決定・アイデア添付／チャット添付 共通）

- **保管先**: **S3 互換オブジェクトストレージ（MinIO）を Docker コンテナで構成**し、1 デプロイ内に含める。会社（テナント）ごとにバケット/プレフィックスで論理分離する。会社 DB には実体ファイルを保存しない。
- **物理ファイル名のハッシュ化（決定）**: ストレージ上の物理オブジェクト名は**ハッシュ等で生成した非可読名**とし、**アップロードされた元のファイル名のままでは保存しない**（名前衝突・情報漏えい・パストラバーサル対策）。
- **DB が保持する情報（決定）**: 会社 DB の Attachment レコードには **①ストレージ上のファイルパス（物理オブジェクトキー）** と **②アップロード時の元ファイル名（表示・ダウンロード時に使用）** の両方を保持する。加えてサイズ・MIME タイプ・アップロード者・アップロード日時等を保持。
- **上限サイズ / 形式**: 1 ファイルの上限サイズと許可形式（画像・PDF・Office 文書 等）を設ける（**具体値は TBD・初期値として設定**）。アップロードはサーバー側でサイズ・MIME を検証する。
- **アクセス制御**: ダウンロードは所属会社・参照権限（当該クエストのパーティー参加）を検証したうえで、署名付き URL 等の一時アクセスで提供する（実装方式の詳細は TBD）。

#### 投票

- 対象: アイデア投票権限を持つ参加メンバー（パーティーメンバー）
- 選択肢: **賛成 / 反対**（1人1票、締切まで変更可能を想定）
- **自分のアイデアへの投票も許可**する
- **投票の匿名化（会社ごとに ON/OFF・決定）**: 投票者を公開するか匿名にするかを、**会社（テナント）単位の設定**で切り替える。設定は会社内の全クエスト・全投票に一律適用する。**既定値は ON（匿名モード）**。
  - **記名モード（匿名化 OFF）**: 誰が賛成/反対したかを公開する。アイデアごとに **賛成したユーザーのアバター** と **反対したユーザーのアバター** を並べて表示（賛成数・反対数もあわせて表示）。
  - **匿名モード（匿名化 ON・既定）**: 投票者の identity を他ユーザーに表示せず、**賛成数・反対数（集計）のみ**を表示する。個々の投票者のアバター・氏名は出さない。ただし**投票者本人には自分の投票内容を表示**し、締切まで変更できる。
  - **匿名モード時の所有者/管理者の扱い（会社ごとに ON/OFF・決定）**: 匿名モードでも**所有者権限・クエスト管理権限を持つユーザーには投票者を開示するか**を会社単位で設定する。
    - **ON（既定）**: 所有者/管理者にも投票者を開示せず、集計数のみ表示（全員に対して匿名を徹底）。
    - **OFF**: 匿名モードでも所有者/管理者だけは投票者（賛成/反対したユーザー）を確認できる（一般メンバーには非表示のまま）。
    - ※この設定は匿名モード（匿名化 ON）のときのみ意味を持つ。記名モードでは全員が投票者を確認できる。
  - どちらのモードでも投票は内部的に記録し（1人1票の制約・XP 付与・変更履歴に使用）、匿名化は**表示上の扱い**のみを制御する。
  - 設定は **システム管理者（運営）が会社の作成・管理の一環として設定**する。
- **XP**: 投票したユーザーに XP（初期値 5）を付与する。軽い行動のため**日次上限の対象**。ただし **評価者による評価には影響しない**（投票結果と評価スコアは独立。第6節ゲーミフィケーション設計参照）。匿名モードでも XP 付与は同じ（付与は本人にのみ紐づく内部処理）

#### チャット

- アイデアごとに 1 つのチャットグループを自動作成
- **参加範囲（決定）**: そのクエストの**パーティーメンバー全員**が閲覧・投稿できる（コメント作成権限を持つメンバー。既定で全参加メンバーに付与）
- メッセージ投稿・履歴表示
- **ファイル添付（決定）**: チャットメッセージにファイルを添付できる。保管・命名方式はアイデア添付と共通（後述「添付ファイルの扱い」参照）
- **メンション（決定）**: `@ユーザー` でパーティーメンバーをメンションできる
- **通知（決定）**: メンション・自分のアイデアへの新規コメント等を通知する。**MVP はアプリ内通知**（ベル/未読）から開始。メール等の外部通知は将来対応（TBD）
- チャット投稿には XP を付与する（軽い行動として日次上限の対象。第6節ゲーミフィケーション設計参照）

#### 評価（評価者による採点）

1 つのアイデアに対し、評価者が以下 5 観点を **各 5 点満点** で採点し、**観点ごとにコメント** を記載する。

| 観点 | 説明 | 配点 |
| --- | --- | --- |
| 新規性 | アイデアの目新しさ・独自性 | 1〜5 |
| 影響度 | 実現した場合のインパクトの大きさ | 1〜5 |
| 実現度 | 実現可能性の高さ | 1〜5 |
| 適合性 | クエストのテーマ・目的への合致度 | 1〜5 |
| コスト | コスト面（**低コストほど高得点**。5点＝非常に低コスト） | 1〜5 |

- 各観点に採点値 + コメント（テキスト）を保持
- 合計/平均点でアイデアを比較・ランキング。**5 観点は均等（重み付けなし・決定）** で単純平均する（コイン付与の平均スコアも同様に均等平均。第6節）。将来重み付けが必要になれば拡張を検討
- **評価者（決定）**: クエストごとに以下のユーザーが評価できる。
  - **クエスト作成者**（デフォルトで評価者）
  - **クエスト作成者が評価者権限を付与したユーザー**（クエスト単位で付与・剥奪）
  - → 評価者は 1 クエストに複数存在しうる（＝1アイデアに複数評価者の評価が付きうる）
- 評価者権限を付与できる対象は **そのクエストの参加メンバーに限定（決定）**。詳細は上記「クエスト内の権限モデル」を参照

## 9. データモデル

（主要エンティティの案。詳細は別ファイル `データモデル.md` に分割予定）

> **DB 構成（2 層）**: エンティティは 2 種類の DB に分かれて格納される。
> - **管理 DB（コントロールプレーン / システム共通・1 つ）**: 全社横断で必要な認証・振り分け情報を持つ。
> - **会社 DB（データプレーン / 会社ごとに物理分離・別コンテナ）**: 各社のクエスト・アイデア等の機密データを隔離して持つ。
> User の識別情報はログイン振り分けのため管理 DB に置き、会社 DB 側にはプロフィール/XP 等の業務データを持つ（同期方法の詳細は TBD）。

### 管理 DB（システム共通）

- **Company（会社 / テナント）**: 名称、接続先会社 DB の識別子（接続情報の実体は `.env` で管理し、キー/参照のみ保持）、状態、**投票匿名化フラグ（vote_anonymized: ON/OFF・会社単位・既定 ON）**、**匿名モード時に所有者/管理者へ投票者を隠すフラグ（hide_voters_from_managers: ON/OFF・会社単位・既定 ON）**、作成日時。※各フラグはログイン時に会社判定と併せて参照し、投票表示（記名/匿名・管理者への開示可否）に反映
- **Account**: ログインアカウント。ログインID、認証情報、所属 Company、システムロール（システム管理者／クエストグループ管理者／一般）
- **AccountQuestGroup**: アカウント × クエストグループの所属（多対多）。※どのクエストグループに属すかの割当はシステム管理者がフル操作。クエストグループ管理者は自グループ範囲のみ
- **SystemAdmin 操作ログ**: 会社作成・DB プロビジョニング等の運用監査（案）

> クエストグループの所属（AccountQuestGroup）は、ログイン時の参照範囲判定に使うため管理 DB 側に保持する想定（会社 DB との配置は実装時に整理・TBD）。

### 会社 DB（会社ごとに分離）

- **User**: ユーザー（会社 DB 内。Account と対応）。ロール（一般／評価者／クエストグループ管理者）、XP、レベル、コイン残高を保持
- **QuestGroup（クエストグループ）**: 会社 DB 内のグルーピング。名称、作成日時。ユーザーは複数所属可
- **QuestGroupMember**: User × QuestGroup の所属（多対多）。パーティー候補・クエスト参照範囲の基礎
- **Quest**: クエスト。**所属クエストグループ（`quest_group_id`）**、作成者（`owner_id`＝既定で所有者・剥奪不可）、カテゴリー、目的/テーマ、期限日、ステータス、作成日時
- **QuestMember（パーティー）**: クエスト参加メンバー（User × Quest の多対多、参加日時を保持）。権限を持つ前提となる所属。**候補は当該クエストの所属クエストグループのメンバーに限定**
- **QuestMemberPermission**: クエスト参加メンバーが持つ権限（QuestMember に対し多）。権限種別（所有者/クエスト管理/評価者/投票/アイデア作成/コメント）、付与者、付与日時。※作成者の所有者権限は剥奪不可
- **Idea**: アイデア。所属クエスト、投稿ユーザー、件名、本文（アイデア）、タイムリミット、利害関係者、価値、備考、作成日時（※会社 DB 内のため会社は暗黙）
- **Attachment**: 添付ファイル（アイデアまたはチャットメッセージに紐づく）。**ストレージ上のファイルパス（物理オブジェクトキー＝ハッシュ名）** と **元のアップロードファイル名**、サイズ、MIME タイプ、アップロード者、アップロード日時を保持。実体は MinIO（S3 互換）に保管し会社ごとに論理分離
- **Vote**: アイデアへの投票（User × Idea）。種別（賛成／反対）、投票日時。1人1票（ユニーク制約）。投票者は常に内部保持（本人の変更・XP 付与に使用）。自分のアイデアへの投票も可。**表示上の記名/匿名は Company の匿名化フラグに従う**（匿名化 ON では他ユーザーに投票者を表示せず集計数のみ）
- **ChatGroup**: アイデアごとのチャットグループ（Idea と 1:1）
- **ChatMessage**: チャット投稿。所属 ChatGroup、投稿者、本文、投稿日時、**添付（Attachment への参照・複数可）**、**メンション対象ユーザー**
- **Notification**: ユーザーへの通知（アプリ内）。宛先 User、種別（メンション／自アイデアへの新規コメント 等）、参照先（Idea/ChatMessage 等）、既読フラグ、作成日時
- **Evaluation**: アイデアへの評価（Idea × 評価者）。評価者、評価日時
- **EvaluationScore**: 評価の観点別スコア（Evaluation に対し多）。観点（新規性/影響度/実現度/適合性/コスト）、点数（1〜5）、コメント
- **Item / Equipment**: 装備アイテム。名称、**スロット（頭/顔/体/手持ち/背景）**、**レアリティ（コモン/標準/レア）**、価格（コイン）、**装備パーツ（VRM 用メッシュ）の参照**（パーツ差し替え方式）
- **UserItem**: ユーザーが所有・装備中のアイテム（多対多）。所有フラグ＋スロットごとの装備中フラグ（各スロット 1 点）。入手経路はコイン購入のみ
- **Transaction / Activity**: XP・コインの付与・消費履歴（不正防止・監査用、案）

## 10. 技術スタック（決定）

- **フロントエンド（決定）**: **Next.js（React）**。3D 表示に **Three.js / React Three Fiber（R3F）**、VRM は **three-vrm** で連携。React エコシステムで R3F/VRM との相性を優先。
- **3D アバター（決定）**: モデル形式は **VRM** を採用（人型アバター特化・ボーン/表情が標準化され着せ替えに有利。**three-vrm** で R3F と連携）。
  - **着せ替え方式（決定）**: **パーツ差し替え方式**。ベースアバターに対し、5 スロット（頭/顔/体/手持ち/背景）ごとの装備パーツ（メッシュ）を取り付け・差し替えする。パーツ追加で装備を拡張しやすい構成とする。
  - 装備パーツは各スロット共通のリグ/取り付け基準に合わせて用意する（TBD: パーツの制作ガイドライン・スキニング方式の詳細）。
- **バックエンド（決定）**: **Python / FastAPI**。軽量・高速で、将来の AI/分析連携にも拡張しやすい。**ログイン時に管理 DB で所属会社を判定し、当該会社 DB へ動的にルーティング**（SQLAlchemy 等でリクエストごとに接続先エンジン/セッションを切り替える構成）。
- **データベース（決定）**: **PostgreSQL**。管理 DB 1 ＋ 会社 DB N を、いずれも Docker コンテナで構成。将来の Kubernetes ＋ Postgres Operator（例: CloudNativePG）移行方針とも整合。
- **インフラ / ホスティング**: **Docker コンテナ**前提。1 デプロイ内に「Web サーバー（リバプロ）＋フロント＋バックエンド＋管理 DB（システム共通）＋会社ごとの DB コンテナ（物理分離）＋**S3 互換オブジェクトストレージ（MinIO）**」を構成。
  - **オブジェクトストレージ（決定）**: 添付ファイルの実体を保管する **MinIO コンテナ**を 1 デプロイに含める。会社（テナント）ごとにバケット/プレフィックスで論理分離し、物理オブジェクト名はハッシュ化する（第6節「添付ファイルの扱い」参照）。※compose 定義は実装フェーズで作成する（現時点で compose ファイルは未作成）。
  - **プロビジョニング方式（MVP・確定）**: **docker compose の単一ファイルを手編集**して会社ごとの DB サービスを追記する（運用コスト最小を優先）。会社追加は「compose に DB サービス追記 → `up` → 管理 DB に接続先を登録」の手動オペレーションとなる。
  - **DB 接続情報（確定）**: `.env` ファイルで管理し、compose から参照（接続情報はリポジトリにコミットしない）。
  - **将来**: テナント数増加・可用性が必要になった段階で、Docker Engine API による自動プロビジョニングや **Kubernetes ＋ Postgres Operator（例: CloudNativePG）** への移行を検討。

- [x] 管理画面 → **システムレベル管理（運営: 会社/DB＋全アカウント・クエストグループ所属をフル操作）** と **クエストグループ管理者（自グループ内アカウントの登録・編集・論理削除）** の 2 階層
- [x] 認証方式 → **独自アカウントで開始し、SSO（SAML/OIDC）連携は将来対応**
- [x] 会社 DB のプロビジョニング方式 → **MVP は docker compose 単一ファイルを手編集して会社 DB を追記**（運用コスト最小）。**DB 接続情報は `.env` で管理**。将来は自動プロビジョニング/k8s＋Operator を検討
- [ ] テナント（会社 DB コンテナ）数の上限・リソース設計
- [ ] 管理 DB と会社 DB 間の User 情報の同期方式
- [x] アイデアの公開範囲 → **そのクエストのパーティー内のみ**（会社全体・グループ全体には非公開。管理者は管理機能の範囲で閲覧）
- [x] 「評価者」の扱い → **クエスト作成者＋作成者が権限付与したユーザー（クエスト単位の権限）。複数評価者を許可**
- [x] 権限付与の対象範囲 → **そのクエストの参加メンバーに限定**
- [x] クエスト内の権限体系 → **所有者/クエスト管理/評価者/投票/アイデア作成/コメント の6権限**（第6節「権限モデル」で定義）
- [x] 新規参加メンバーの既定権限 → **投票＋アイデア作成＋コメントを自動付与**（他は手動付与）
- [x] メンバーをパーティーから外した際のデータ → **既存の入力（アイデア/投票/評価/コメント）は削除せず表示継続**。権限のみ失う
- [x] クエストのカテゴリー → **事前定義リストからの選択＋自由入力の併用**
- [x] 投票の仕様 → **自分のアイデアへの投票可・投票者にXP付与（初期値5・日次上限）・評価には非影響**
- [x] 投票の匿名化 → **会社（テナント）単位で ON/OFF 設定・既定 ON（匿名）**。記名モードは賛成/反対のアバターを並べて表示、匿名モードは集計数のみ表示（投票は内部保持し本人の変更・XP付与に使用）。加えて**匿名モード時に所有者/管理者へ投票者を隠すか**も会社単位で設定・既定 ON。設定はシステム管理者が会社管理の一環で行う
- [x] チャット参加範囲 → **パーティーメンバー全員が閲覧・投稿可**
- [x] チャットの添付・メンション・通知 → **すべて対応**。添付は共通の添付方式（MinIO・ハッシュ名）／`@` メンション／通知は MVP はアプリ内通知（外部通知は将来）
- [x] 評価「コスト」観点の採点方向 → **低コストほど高得点**（5点＝非常に低コスト）
- [x] 5 観点の重み付けの有無 → **均等（重み付けなし）で単純平均**。将来必要になれば拡張を検討
- [x] アイデア各項目の必須/任意 → **件名・アイデア本文・価値を必須**。タイムリミット・利害関係者・備考・添付は任意
- [x] 添付ファイルの保管先・命名・保持情報 → **S3 互換オブジェクトストレージ（MinIO）を Docker コンテナで構成**。物理ファイル名はハッシュ化（元名で保存しない）。**DB はファイルパス＋元ファイル名を保持**。会社ごとにバケット/プレフィックスで論理分離
- [ ] 添付ファイルの上限サイズ・許可形式の具体値（初期値として設定）
- [x] XP・コインの具体的な付与ルールとレベル設計 → **XP は幅広く付与（成果重視カーブ）＋日次上限で乱獲対策 / コインは評価点の平均スコア連動で希少付与 / レベルは総量式の線形上昇**（第6節ゲーミフィケーション設計に反映。数値は初期値・調整可）
- [x] 装備アイテムの種類・入手方法 → **5 スロット（頭/顔/体/手持ち/背景・各1点）** ／ **コイン購入のみ** ／ **レアリティ3段階の幅広い価格帯（コモン10〜30・標準50〜150・レア300〜800）**（第6節に反映・初期値）
- [x] 3D アバターのモデル形式・着せ替え実現方式 → **VRM 採用（three-vrm で R3F 連携）＋パーツ差し替え方式**（5 スロットごとに装備メッシュを取り付け・差し替え）
- [x] 技術スタックの確定 → **フロント: Next.js（React）＋R3F/three-vrm ／ バックエンド: Python・FastAPI（会社DBへ動的ルーティング） ／ DB: PostgreSQL（管理DB＋会社DB N・Docker）**
- [ ] マネタイズの有無（コイン課金など）

## 13. 用語集

| 用語 | 定義 |
| --- | --- |
| ideaquest | 本プロジェクトの名称 |
| 会社（テナント） | データ分離の単位。1 デプロイに複数同居し、会社ごとに DB を物理分離 |
| クエストグループ | 会社 DB 内のグルーピング。ユーザーは複数所属可。クエストはいずれかのグループに属し、パーティー候補・参照範囲の基礎になる |
| 管理 DB（コントロールプレーン） | システム共通の DB。認証・アカウント・所属会社・クエストグループ所属・会社 DB 接続情報を一元管理 |
| 会社 DB（データプレーン） | 会社ごとに物理分離した DB（別コンテナ/インスタンス）。クエスト・アイデア等の機密データを隔離 |
| オブジェクトストレージ（MinIO） | 添付ファイルの実体を保管する S3 互換ストレージ（Docker コンテナ）。会社ごとにバケット/プレフィックスで論理分離し、物理ファイル名はハッシュ化する |
| システム管理者 | サービス運営側。会社の作成・DB プロビジョニング、全アカウントとクエストグループ所属をフル操作 |
| クエストグループ管理者 | 自分が所属するクエストグループ内のアカウントを登録・編集・論理削除できる管理者 |
| クエスト | アイデア出しのテーマ単位。カテゴリー・目的/テーマ・期限・参加メンバーを持つ |
| パーティー / パーティーメンバー | クエストに参加するメンバーの集まり・各メンバー |
| 投票 | アイデアに対する賛成／反対の意思表示 |
| 投票匿名化 | 投票者を表示するか（記名モード）匿名にするか（匿名モード＝集計数のみ表示）を切り替える設定。**会社（テナント）単位**で ON/OFF する（既定 ON）。匿名モード時に所有者/管理者へ投票者を開示するかも会社単位で別途設定できる（既定は非開示） |
| チャットグループ | アイデアごとに作られる、メンバー間の意見交換の場 |
| 評価者 | アイデアを 5 観点で採点・コメントする役割のユーザー（評価者権限を持つ参加メンバー） |
| 評価観点 | 新規性・影響度・実現度・適合性・コストの 5 項目（各 5 点満点） |
| 所有者権限 | クエストの全操作が可能な権限。作成者は既定で所有者（剥奪不可）。他者への付与は作成者のみ |
| クエスト管理権限 | クエストの編集（参加メンバー調整・権限付与含む）が可能な権限 |
| XP（経験値） | 行動に応じて付与されるポイント。累積でレベルが上がる |
| レベル | XP 累積に応じたユーザーの成長度合い |
| コイン | ショップで装備を購入するための通貨 |
| 装備 / アイテム | 3D アバターに着せ替えできる要素 |
| アバター | ユーザーを表す 3D キャラクター |
