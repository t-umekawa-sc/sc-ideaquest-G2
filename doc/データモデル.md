## 1. 全体構成（2 層 DB）

エンティティは **2 種類の DB** に分かれて格納される（README 第9節）。

- **管理 DB（コントロールプレーン / システム共通・1 つ）**
  認証・テナント振り分けに必要な情報を全社横断で保持。ログイン時にここで所属会社を判定し、当該 **会社 DB** へ動的ルーティングする。
- **会社 DB（データプレーン / 会社ごとに物理分離・別コンテナ）**
  各社のクエスト・アイデア・チャット・ゲーミフィケーション等の機密データを隔離して保持。

```mermaid

  subgraph CP["管理 DB（システム共通・1つ）"]
    companies["companies（会社）"]
    accounts["accounts（アカウント）"]
    trusted_devices["trusted_devices（信頼済み端末）"]
    otp_challenges["otp_challenges（OTPチャレンジ）"]
    system_audit_logs["system_audit_logs（監査ログ）"]
    account_sync_outbox["account_sync_outbox（会社DB同期）"]
  end
  subgraph DP1["会社 DB #1（テナントA・物理分離）"]
    UserA["users / quests / ideas / chat_messages ..."]
  end
  subgraph DP2["会社 DB #N（テナントN・物理分離）"]
    UserN["users / quests / ideas / chat_messages ..."]
  end
  accounts -->|company_id| companies
  companies -.->|接続先識別子→.env| DP1
  companies -.->|接続先識別子→.env| DP2
  accounts -.->|account_id（論理対応・DB跨ぎFKなし）| UserA
```

**DB 跨ぎの参照は物理 FK を張れない**（別インスタンスのため）。管理 DB の `accounts` と 会社 DB の `users` は **`account_id`（UUID）で論理的に対応付け**る。整合はアプリ層で担保し、**認証系属性（`login_id`/`email`/`status`）は `accounts` を源泉に `users` へ一方向ミラー**する（アウトボックス＝§4.6 `account_sync_outbox`・冪等/結果整合）。これによりユーザ一覧は会社 DB 単独で描画できる（§5.3・§8-①）。

## 2. 命名規約・共通方針

- **物理テーブル名**: スネークケースの **複数形**（例: `quests`, `idea_revisions`, `evaluation_scores`）。
- **論理名（日本語）**: 各テーブル定義の見出しに「物理テーブル名（複数形 snake_case）＋日本語論理名」を、各カラムに「物理名＋日本語論理名」を併記する。ER 図のエンティティ名も物理テーブル名（複数形）に揃える。
- **カラム名**: スネークケース（英小文字）。
- **主キー**: `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`。
  - 理由: マルチテナント・列挙耐性（連番推測を避ける）・MinIO オブジェクトキーや URL への露出耐性。`pgcrypto`（or PG13+ の `gen_random_uuid()`）前提。
- **タイムスタンプ**: `timestamptz`（UTC 保存・表示層で JST 変換）。
- **共通監査カラム（全テーブル共通装備）**: どのレコードも「**だれが・いつ・どのプログラムで**」登録／最終更新したかを追えるよう、**登録3点**（`created_at`/`created_by_id`/`created_program`）と**更新3点**（`updated_at`/`updated_by_id`/`updated_program`）を **全テーブルに必ず持たせる**。詳細は **§2.1**。
- **論理削除**: アカウントは無効化（`status=disabled`）＝論理削除（入力履歴は残す）。その他は原則物理削除しない設計を優先（監査・履歴保持）。
- **外部キー**: 会社 DB 内は物理 FK を張る。**DB 跨ぎ**（accounts↔users＝`account_id` の論理対応）は **論理参照**（FK なし・アプリ整合）。**FK カラムの命名規則は §2.2** に定める。
- **Enum**: PostgreSQL の `ENUM` 型 or `text + CHECK`。本書では値集合を §3 に集約。実装ではマイグレーション容易性から `text + CHECK` を推奨（値追加が軽い）。
- **金額/ポイント**: XP・コイン・SP は `integer`（残高系は 0 以上を保証＝`CHECK (>= 0)`）。
- **添付/画像パス**: MinIO オブジェクトキー（ハッシュ名）を `text` で保持。バケット/プレフィックスは会社ごとに論理分離（実体名は元ファイル名と別）。

### 2.1 共通監査カラム（全テーブル共通装備）

**方針**: **すべてのテーブル**（管理 DB・会社 DB とも）に、以下の **6 カラムを共通装備**する。目的は「**どのレコードも、だれがいつどのプログラムで登録し、最後に更新したかが分かる**」こと。

| 物理名 | 論理名 | 型 | 制約 / 既定 | 説明 |
| --- | --- | --- | --- | --- |
| `created_at` | 登録日時 | timestamptz | NOT NULL, default now() | レコード作成時刻（UTC） |
| `created_by_id` | 登録者ID | uuid | NULL 可 | 登録した操作者。**会社 DB→`users`／管理 DB→`accounts`**。システム/バッチ登録は NULL（`created_program` で識別） |
| `created_program` | 登録プログラム | text | NOT NULL | 登録元プログラム識別子（例: API エンドポイント名・バッチ名・画面 ID `SC-xx`・`migration`・`seed`） |
| `updated_at` | 更新日時 | timestamptz | NOT NULL, default now() | **更新の都度** now() に更新（作成時は登録日時と同値） |
| `updated_by_id` | 更新者ID | uuid | NULL 可 | 最終更新の操作者。参照先は `created_by_id` と同じ。システム/バッチ更新は NULL |
| `updated_program` | 更新プログラム | text | NOT NULL | 最終更新元プログラム識別子（作成時は `created_program` と同値） |

- **命名**: `created_by_id`/`updated_by_id` は §2.2 の操作者規則（`<動作>_by_id`）に従う。プログラム列は `_program` サフィックス（`text`）。
- **参照先**: `created_by_id`/`updated_by_id` は会社 DB では `users`、管理 DB では `accounts` を指す。会社 DB 内は物理 FK（`ON DELETE RESTRICT`）、`accounts` を跨ぐ管理 DB 側は原則同一 DB 内 FK。**利用者を伴わない登録**（メール OTP 発行・監査ログ・プロビジョニング等）は `*_by_id` を NULL とし、`*_program` に処理主体を必ず残す。
- **更新の担保**: `updated_at`/`updated_by_id`/`updated_program` はアプリ層で更新の都度必ずセットする（または `BEFORE UPDATE` トリガで `updated_at` を強制）。監査ログ（`system_audit_logs`）や版管理（`idea_revisions`）は変更内容の履歴、本 6 カラムは各行の最終状態のメタ、と役割を分ける。
- **記載の省略**: 本 6 カラムは全テーブル共通のため、以降の各テーブル定義（§4・§5）および ER 図では **一律に列挙を省略**する（6 カラムは本規約によりすべてのテーブルに自動付与される）。各定義表・ER 図に監査カラムは記載しない。なお `created_at` を並び順・集計キーに使うインデックス定義（例: `(chat_group_id, created_at)`）は、カラムが本規約で存在することを前提にそのまま記す。

### 2.2 外部キー（FK）の命名規則

**基本規則**: FK カラム名 ＝ **`<参照先テーブル名の単数形>_id`**。

- 例: `company_id`→`companies` / `quest_id`→`quests` / `idea_id`→`ideas` / `evaluation_id`→`evaluations` / `chat_group_id`→`chat_groups`。
- **単数化**: 複数形を素直に単数化する。不規則・複合語は以下に従う。
  - `companies`→`company`（`company_id`）、`activities`→`activity`（`activity_id`）
  - `quest_categories`→`quest_category`、`idea_stakeholders`→`idea_stakeholder`、`chat_messages`→`chat_message`（複合語は末尾のみ単数化）

**例外規則（役割名を付ける場合）**: 参照先が一意に決まらない／意味を明確にしたい場合は **役割名** を用いる。命名は原則 `<役割>_<参照先単数形>_id`、慣用的に参照先が自明なものに限り `<役割>_id` を許容し、**定義表に参照先（`FK→<table>`）を必ず明記**する。

1. **意味的な役割 / 同一テーブルへ複数 FK**:
   - `<役割>_id`（参照先が自明なもの）: `owner_id` / `author_id` / `editor_id` / `evaluator_id` / `recipient_id`（いずれも→`users`）
   - `<役割>_<単数形>_id`（明確化）: `mentioned_user_id`（→`users`）、`actor_account_id`（→`accounts`）
2. **自己参照**: `<役割>_<単数形>_id`。例: `requires_spell_id`（→`spells`）、`reply_to_message_id`（→`chat_messages`）
3. **操作者（誰が〜したか）**: `<動作>_by_id`（**末尾は必ず `_id`**）。例: `uploaded_by_id`（→`users`）、`granted_by_id`（→`users`）
4. **任意・多態的な参照**（1 カラムで種別により参照先が変わり得る／複数の参照候補を並べる通知等）: 接頭辞 `ref_` ＋ `<単数形>_id`。例: `ref_idea_id` / `ref_chat_message_id` / `ref_idea_revision_id` / `ref_achievement_id`。型を持たず ID のみの汎用参照は `ref_id`（FK なし・説明で参照先を明記）。

**共通ルール**:

- FK カラムは **必ず `_id` で終える**（`_by` 単独等は使わない）。
- **参照先は定義表の `FK→<table>` 表記または説明で必ず明示**（役割名から参照先が読み取れない場合の曖昧さ回避）。
- **DB 跨ぎの参照**（`accounts`↔`users` の `account_id` 対応）は物理 FK を張らない **論理参照**。カラム名は同一規則に従い、定義表に「論理参照→…」と明記する。
- 会社 DB 内は物理 FK を張り、`ON DELETE` は原則 **`RESTRICT`**（履歴保持のため物理削除を避け、論理削除/トゥームストーンを優先）。

### 4.1 companies（会社 / テナント）

会社（テナント）。**会社単位の設定フラグ**（投票匿名化・管理者への開示・MFA 必須）を保持し、ログイン時に会社判定と併せて参照する（SC-92 会社設定で編集）。

| 物理名 | 論理名 | 型 | 制約 / 既定 | 説明 |
| --- | --- | --- | --- | --- |
| `id` | ID | uuid | PK | |
| `company_code` | 会社コード | text | NOT NULL, UNIQUE | 業務上の会社識別コード（全社で一意）。ログイン時の会社指定・管理表示に使用（`db_identifier` とは別の対人向けコード） |
| `name` | 会社名 | text | NOT NULL | |
| `db_identifier` | 会社DB識別子 | text | NOT NULL, UNIQUE | 会社 DB の識別子（接続情報の実体は `.env`。ここはキー/参照のみ） |
| `status` | 状態 | company_status | NOT NULL, default `active` | active / suspended（メンテ中） |
| `color` | 会社カラー | text | NOT NULL, default（既定色） | プリセットパレットの hex（クエストカラーと同方式） |
| `icon_image_path` | 会社アイコン画像パス | text | NULL 可 | 会社アバター/アイコン（MinIO キー・任意）。未設定時は「頭文字＋会社カラー」タイル |
| `vote_anonymized` | 投票匿名化フラグ | boolean | NOT NULL, default `true` | ON=集計数のみ表示 |
| `hide_voters_from_managers` | 管理者への投票者非開示フラグ | boolean | NOT NULL, default `true` | 匿名時に所有者/管理者へも投票者を隠す |
| `mfa_required` | MFA必須フラグ | boolean | NOT NULL, default `true` | メール OTP による MFA を必須化 |

- インデックス: `UNIQUE(company_code)`、`UNIQUE(db_identifier)`、`(status)`。
- **会社コードのフォーマット規約（初期値・調整可）**: 半角英大文字・数字・ハイフン（`A–Z` / `0–9` / `-`）、**4〜20 文字**、**先頭は英字**、**全社で一意**、**大小文字は区別しない**（入力を大文字へ正規化して保存・照合）。原則**作成時に確定し以後不変**（対人向け識別子のため変更は運用影響大）。
- 注: `hide_voters_from_managers` は `vote_anonymized=false`（記名モード）のとき意味を持たない（アプリ側で無効化表示・SC-92）。

### 4.2 accounts（アカウント）

**ログイン ID とメールは別カラム**（README 2026-07-19 変更）。ログイン ID は英数字等の任意識別子でも可、メール変更でログイン ID は不変。

| 物理名 | 論理名 | 型 | 制約 / 既定 | 説明 |
| --- | --- | --- | --- | --- |
| `id` | ID | uuid | PK | 会社 DB の `users.account_id` と論理対応 |
| `company_id` | 所属会社ID | uuid | NOT NULL, FK→companies | 所属会社（テナント） |
| `login_id` | ログインID | text | NOT NULL | ログイン識別子。**会社内で一意**（`UNIQUE(company_id, login_id)`） |
| `email` | メールアドレス | text | NOT NULL | OTP 送信・初回パスワード設定リンク・将来の外部通知の宛先。**会社内で一意**（`UNIQUE(company_id, email)`） |

### 4.6 account_sync_outbox（会社DB同期アウトボックス）

`accounts` の `login_id`/`email`/`status`（および無効化・氏名等の付随情報）を**会社 DB `users` へ一方向ミラー**するためのアウトボックス。**アカウント発行/編集/無効化/PWリセット、および本人のプロフィール編集**で `accounts` を更新するのと**同一トランザクション**で 1 行 INSERT し、常駐ワーカ（または定期ジョブ）が対象会社 DB の `users` へ冪等に適用する（会社 DB は別インスタンスのため 2 相コミットは行わず、アウトボックス＋再試行で結果整合を担保）。

| 物理名 | 論理名 | 型 | 制約 / 既定 | 説明 |
| --- | --- | --- | --- | --- |
| `id` | ID | uuid | PK | |
| `account_id` | アカウントID | uuid | NOT NULL, FK→accounts | 対象アカウント |
| `company_id` | 会社ID | uuid | NOT NULL, FK→companies | 反映先会社 DB の特定に使用 |
| `op` | 操作種別 | sync_op | NOT NULL | `upsert`（発行/編集）/ `disable`（無効化）/ `enable`（再有効化） |
| `payload` | 反映内容 | jsonb | NOT NULL | `users` へ反映する値（`login_id`/`email`/`status`/`display_name` 等）。会社 DB 側で **`account_id` をキーに upsert**（冪等） |
| `status` | 同期状態 | sync_status | NOT NULL, default `pending` | `pending` / `done` / `failed` |
| `attempts` | 試行回数 | int | NOT NULL, default 0 | 失敗時のリトライ回数（上限超で `failed`＝要手動対応） |
| `processed_at` | 処理日時 | timestamptz | NULL 可 | 反映完了時刻 |

- インデックス: `(status, id)`（未処理の取り出し順）、`(account_id)`、`(company_id, status)`。
- **冪等性**: 会社 DB 側は `account_id` をキーに upsert（同一イベントの再適用が安全）。反映先 `users` 行が未作成でも upsert で作成される（プロビジョニング直後の初回同期を兼ねる）。
- **順序**: 同一 `account_id` は `id`（生成順）で直列適用。
- `otp_challenges` 同様、実体を Redis/ジョブキューにする案もあるが、確実性重視で管理 DB にアウトボックステーブルを持つ本方式を基本とする（§8-①）。

## 6. 全文検索の索引（FR-31）

- 対象（会社 DB）: `ideas`（`title`/`body`/`value`/`note`）＋ `chat_messages`（`body`）。

## 7. 集計・派生値の扱い（XP/コイン/SP/ランキング）

数値は初期値・調整可（README 第6節）。

- **XP 付与**: 自アイデア選定=200 / アイデア投稿=50 / 評価実施=30 / ログイン=10 / 投票=5 / チャット=5。軽い行動（投票/チャット/ログイン）は **日次上限**（上限値 TBD §8）。`activities(kind=xp_gain)` に記録し、`users.xp` を更新。
- **レベル**: Lv n→n+1 必要 XP = `100+(n-1)×50`（総量式の線形上昇）。`users.level` は `xp` の従属キャッシュ。レベルアップで **SP+1**（`activities(kind=sp_gain, reason=levelup_sp)`＋`users.skill_point_balance` 更新）。
- **コイン**: 行動では付与せず、**評価点連動のみ**（希少）。全評価者×全 5 観点の均等平均 `avg` に対し `round(avg × 10)`（最大 50/アイデア）を投稿者へ付与。**加えて実績獲得ボーナス**（ティア連動 20/50/150）。付与の確定タイミング（評価締切一括 or 都度再計算）は TBD（§8）。
- **SP 消費**: 魔法解放（`user_spells` 追加＋`activities(kind=sp_spend)`＋残高減算）。前提魔法の解放済みが必要（スキルツリー）。
- **コイン消費**: ショップ購入（`user_items` 追加＋残高検証＋減算＋`activities(kind=coin_spend)`）。サーバー側で残高・価格を検証。
- **ランキング**（SC-41 / ダッシュボード週間 / クエスト内週間）: スコア＝期間内の **獲得 XP＋獲得コイン**（`activities` から集計）。SP は対象外。クエスト内ランキングは `activities.quest_id` で絞る。週起点/TZ/同点タイブレークは TBD（§8）。
- **実装方針**: 残高（`users.xp/coin_balance/skill_point_balance`）は **キャッシュ**で持ちつつ、真実は `activities` 元帳。整合性のため付与/消費は必ず `activities` 追記＋残高更新を **同一トランザクション**で行う。

1. **管理 DB ↔ 会社 DB の users 同期 ＝ 決定（2026-07-20）**: `accounts.login_id`/`email`/`status` を会社 DB `users` に**一方向ミラー**し、ユーザ一覧は会社 DB 単独で描画（`WHERE status='active'`・API 往復なし）。**源泉＝`accounts`**（一意検証もここ。`login_id`・`email` は**会社内で一意**）。反映は §4.6 `account_sync_outbox`＋ワーカで冪等・結果整合。変更契機＝管理者操作（SC-90/92）＋本人のプロフィール編集（`login_id`/`email`）。**クエストグループ所属の配置＝決定（2026-07-20）**: 会社 DB `quest_group_members`（§5.5）を**唯一の正**とし、管理 DB の `account_quest_groups` は**廃止**（ログイン時の参照範囲判定は会社 DB 接続後に実施＝管理 DB 保持は不要。二重管理回避・機密性重視。全社横断のQG参照は当面対象外）。割当操作（SC-90/SC-92）は対象会社 DB に対して行う。
2. **ログイン時の会社特定方法 ＝ 決定（2026-07-20）**: **全社同一 URL＋ログイン画面で会社コード（`companies.company_code`）を手入力**して会社を特定（サブドメイン方式・login_id グローバル一意方式は不採用）。`accounts` の一意制約は現状維持（`UNIQUE(company_id, login_id)`／`UNIQUE(company_id, email)`）。フロー＝会社コードで `company_id`＋`db_identifier` を解決 → 会社内で `login_id`＋`password` 検証 → 会社別 MFA（メールOTP・信頼端末スキップ）→ 当該会社 DB へルーティング。**エラーは会社コード/ID/PW の誤りを区別せず一様**（列挙耐性）。会社コードは端末記憶でプリフィル可（信頼端末と併用）。
3. **マスタ（spells / items / achievements）の配置 ＝ 決定（2026-07-20）**: **各会社 DB に同一シードを配布**（管理 DB 共通マスタ＋DB 跨ぎ参照は不採用＝クロス DB 結合を避ける／中央マスタ＋複製も MVP には過剰）。各マスタに安定コード `code`（UNIQUE）を追加し、**バージョン管理したシードをプロビジョニング時＋リリース時に `code` upsert で冪等適用**（スキーマ migration と同一パイプライン）。`user_*`・`reactions` は同一 DB 内 FK。**全社同一カタログ**（会社ごとカスタムは当面対象外）。
