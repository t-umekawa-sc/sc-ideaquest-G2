> 画面遷移図: `doc/画面設計/画面遷移図.md` ／ モック: `doc/画面設計/mocks/SC-00_ログイン.html` ／ 要件: `doc/要件定義/README.md`（FR-03 認証・FR-05 ログインボーナス）

## 1. 概要・目的

- 独自アカウントで **ログイン／ログアウト** を行う入口画面（未認証者向け）。
- **アカウントは管理者発行のみ**（自己新規登録は無い）。一般ユーザーは**初回ログイン時にパスワード設定**を行う。
- **多要素認証（MFA）**: ID＋パスワード（第1要素）成功後、**メールで認証コード（OTP・第2要素）**を送信し、その入力成功でログインを許可。**会社ごとに MFA の ON/OFF（既定 ON）**。
- **信頼デバイス**: OTP入力時に「このデバイスを信頼する」ON で、次回以降その端末では **OTP をスキップ（有効期限30日）**。別端末では常に OTP を要求。**セッション保持はデバイス信頼に一本化**（別途の「ログイン状態を保持する」は設けない）。
- ログイン成功で所属会社を判定し、当該会社DBへ接続してダッシュボード（SC-01）へ遷移。**ログインで XP 付与（ログインボーナス・1日1回・日次上限対象）**。

## 2. アクセス条件・権限

| 項目 | 内容 |
| --- | --- |
| アクセス可能者 | 全員（未認証）。認証済みでこの画面に来た場合は SC-01 へリダイレクト |
| ロール | 不問（ログイン後にシステムロール＝一般／クエストグループ管理者／システム管理者を判定） |
| テナント | **会社選択欄は無し**。ログインIDから管理DBで所属会社を判定（クロステナント選択をユーザーに委ねない） |

## 3. 画面状態

本画面は3状態を持つ（モックではデモ切替）。

1. **A: ログイン** — ID＋パスワードでの第1要素認証。
2. **C: 認証コード入力（MFA）** — メール送信されたOTPを入力（第2要素）。「このデバイスを信頼する」チェックあり。
3. **B: 初回パスワード設定** — 初回ログイン（仮パスワードでの認証成功かつ未設定）時に表示。新パスワードを設定して利用開始。

> 標準フロー: A（ID＋PW）→（信頼端末なら C をスキップ）→ C（OTP）→（初回なら B）→ SC-01。
> 初回フロー: 管理者発行の**仮パスワード**でAからログイン → OTP（C）→ バックエンドが「要パスワード変更」を返す → B → 設定完了で SC-01。
> MFA=OFF の会社: C を経由せず A（→初回なら B）→ SC-01。

## 5. アクション・遷移

| トリガー | 条件 | 遷移/結果 |
| --- | --- | --- |
| ログイン押下 | 第1要素成功・MFA=OFF or 信頼端末Cookie有効 | OTPスキップ。パスワード設定済なら SC-01（ログインボーナスXP）、要変更なら状態B |
| ログイン押下 | 第1要素成功・MFA必要（信頼端末でない） | **メールにOTP送信** → 状態C（認証コード入力）へ |

## 6. バリデーション

- **ログインID**: 必須・メール形式。空/形式不正はクライアント側で指摘。
- **パスワード（ログイン）**: 必須（強度チェックはしない＝既存値の照合のみ）。
- **認証コード（OTP）**: 必須・6桁数字。期限内・単回・試行上限内であることをサーバーが判定。
- **新パスワード**: 必須・**ポリシー準拠**（最小長・文字種など。値はTBD＝要件で確定）。
- **新パスワード（確認）**: 新パスワードと**完全一致**。
- サーバー側でも同一の検証を行う（クライアント検証は補助）。認証・所属会社判定・仮パスワード期限などはサーバーが最終判断。

## 8. API 入出力（案・API設計フェーズで確定）

> 実際のパス/認証方式（トークン or セッションCookie）はAPI設計で確定。以下は設計の当たり。

- **POST `/api/auth/login`**（第1要素）
  - in: `{ login_id, password }`（信頼デバイスCookieがあれば送信される）
  - out(MFA不要＝OFF or 信頼端末・設定済): `{ status: "ok", user, role, company_id }`＋セッション（HttpOnly Cookie）
  - out(要OTP): `{ status: "mfa_required", mfa_token, masked_email }` → メール送信済 → 状態C
  - out(要変更): `{ status: "password_change_required", change_token }`
  - out(失敗): `401 { error_code }`（理由は最小限）
- **POST `/api/auth/mfa/verify`**（第2要素）
  - in: `{ mfa_token, code, trust_device }`
  - out(成功): `{ status: "ok" | "password_change_required", ... }`＋セッション。`trust_device=true` なら**信頼デバイスCookie（HttpOnly/Secure・30日）**を発行し TrustedDevice を登録
  - out(失敗): `{ error_code: invalid_code | expired | too_many_attempts }`
- **POST `/api/auth/mfa/resend`**: `{ mfa_token }` → 新OTP送信（クールダウン/上限をサーバー判定）。
- **POST `/api/auth/password/initial`**（初回/リセット後の設定）
  - in: `{ change_token, new_password }`
  - out: `{ status: "ok" }`＋セッション → SC-01
- **POST `/api/auth/logout`**: セッション破棄（信頼デバイスCookieは任意で保持/失効）。
- 補足: OTP生成・送信・照合、信頼デバイス照合、ログインボーナスXP付与（1日1回・日次上限）、所属会社判定→会社DB接続はすべてサーバー側。MFA要否は Company の `mfa_required` を参照。

## 9. 非機能・セキュリティ

- パスワードは**ハッシュ保存**（アルゴリズムは実装で確定）。通信は HTTPS 前提。
- 認証失敗の**理由を詳細に返さない**（アカウント存在の推測を防ぐ）。
- **総当り対策**: ログイン/OTP双方に試行回数制限・一時ロック・レート制限（閾値はTBD）。
- **MFA（メールOTP）**: 6桁数字・有効期限10分・単回使用・試行上限（例5回）・再送クールダウン60秒/回数上限。OTPは**ハッシュで保持**し平文で残さない。
- **信頼デバイス**: 暗号乱数トークンを**ハッシュ保存**、Cookieは HttpOnly/Secure/SameSite=Lax。有効期限30日。パスワード変更/リセット・全端末サインアウト・管理者操作で失効。**信頼はOTPスキップのみを意味し、パスワードは毎回必要**。
- **セッション**: デバイス信頼に一本化。**信頼端末はセッションも長期（〜信頼期限のローリング）／非信頼端末は短命セッション**（具体値はTBD）。
- **残存リスク**: ログインIDとメールは別項目にしたが、OTP 配送がメールのため登録メール侵害はログイン突破の一要因になり得る（将来 TOTP 追加を検討）。
- **クロステナントアクセス禁止**: 会社DB接続はログインで確定した所属会社に限定。
- CSRF/XSS 対策、Cookie 属性（HttpOnly/Secure/SameSite）は実装で担保。
