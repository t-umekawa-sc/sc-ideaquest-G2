## 1. 概要・目的

- 独自アカウントで **ログイン／ログアウト** を行う入口画面（未認証者向け）。
- **アカウントは管理者発行のみ**（自己新規登録は無い）。一般ユーザーは**初回ログイン時にパスワード設定**を行う。
- ログイン成功で所属会社を判定し、当該会社DBへ接続してダッシュボード（SC-01）へ遷移。**ログインで XP 付与（ログインボーナス・1日1回・日次上限対象）**。

## 2. アクセス条件・権限

| 項目 | 内容 |
| --- | --- |
| アクセス可能者 | 全員（未認証）。認証済みでこの画面に来た場合は SC-01 へリダイレクト |
| ロール | 不問（ログイン後にシステムロール＝一般／クエストグループ管理者／システム管理者を判定） |
| テナント | **会社選択欄は無し**。ログインIDから管理DBで所属会社を判定（クロステナント選択をユーザーに委ねない） |

## 3. 画面状態

本画面は2状態を持つ（モックではデモ切替）。

1. **A: ログイン** — ID＋パスワードでの通常ログイン。
2. **B: 初回パスワード設定** — 初回ログイン（仮パスワードでの認証成功かつ未設定）時に表示。新パスワードを設定して利用開始。

> 初回フロー: 管理者発行の**仮パスワード**でAからログイン → バックエンドが「要パスワード変更」を返す → Bへ遷移 → 設定完了で SC-01 へ。

| トリガー | 条件 | 遷移/結果 |

## 6. バリデーション

- **ログインID**: 必須・メール形式。空/形式不正はクライアント側で指摘。
- **パスワード（ログイン）**: 必須（強度チェックはしない＝既存値の照合のみ）。
- **新パスワード**: 必須・**ポリシー準拠**（最小長・文字種など。値はTBD＝要件で確定）。
- **新パスワード（確認）**: 新パスワードと**完全一致**。
- サーバー側でも同一の検証を行う（クライアント検証は補助）。認証・所属会社判定・仮パスワード期限などはサーバーが最終判断。

## 8. API 入出力（案・API設計フェーズで確定）

> 実際のパス/認証方式（トークン or セッションCookie）はAPI設計で確定。以下は設計の当たり。

- **POST `/api/auth/login`**
  - in: `{ login_id, password, remember }`
  - out(成功・設定済): `{ status: "ok", user: {...}, role, company_id }`＋認証情報（HttpOnly Cookie 想定）
  - out(要変更): `{ status: "password_change_required", change_token }`
  - out(失敗): `401 { error_code }`（理由は最小限）
- **POST `/api/auth/password/initial`**（初回/リセット後の設定）
  - in: `{ change_token, new_password }`
  - out: `{ status: "ok" }`＋認証情報 → SC-01
- **POST `/api/auth/logout`**: セッション破棄。
- 補足: ログインボーナスXP付与はログイン成功時にサーバー側で判定（1日1回・日次上限）。所属会社判定→会社DB接続もサーバー側。

## 9. 非機能・セキュリティ

- パスワードは**ハッシュ保存**（アルゴリズムは実装で確定）。通信は HTTPS 前提。
- 認証失敗の**理由を詳細に返さない**（アカウント存在の推測を防ぐ）。
- **総当り対策**: 試行回数制限/一時ロック/レート制限（閾値はTBD）。
- 「ログイン状態を保持する」= セッション有効期間の延長。**機密データを扱う社内アプリのため既定OFF**・共有端末での利用に注意喚起（有効期間の具体値はTBD）。
- **クロステナントアクセス禁止**: 会社DB接続はログインで確定した所属会社に限定。
- CSRF/XSS 対策、Cookie 属性（HttpOnly/Secure/SameSite）は実装で担保。
